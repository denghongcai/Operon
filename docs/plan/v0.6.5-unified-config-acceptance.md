# v0.6.5 Unified Config Acceptance

## Goal

v0.6.5 replaces the split CLI/daemon setup files with one entrypoint config.

```text
v0.6.5 = config.yaml as the single configuration entrypoint.
```

## Scope

- Add an [`operon-config`](../../crates/operon-config) crate for shared config schema and loading.
- Make `operon --config <path>` read `client.nodes` from unified config.
- Make `operond start --config <path>` read `daemon`, `policy`, `auth`,
  `store`, and `secrets` references from unified config.
- Default missing `--config` to `$HOME/.operon/config.yaml`.
- Update onboard to generate `.operon/config.yaml`.
- Keep daemon/client sections separate inside the same YAML file.
- Keep sensitive values as references through `token_file`, `token_env`, or
  `secrets.file` where appropriate.

## Non-goals

- No legacy `nodes.yaml` compatibility path.
- No legacy daemon flag compatibility path for policy/auth/store/secrets.
- No inline secret-value recommendation.
- No remote policy mutation.

## Done When

- `operon` and `operond` can both use the same config schema.
- `operon onboard` writes `config.yaml` instead of split config files.
- Validation scripts use unified config for both daemon and client.
- Docs present unified config as the only supported format.
