# v0.13.6 Test Hardening

Status: Completed.

## Goal

Turn the recent test-coverage review into targeted, high-signal coverage for
the parts of Operon that are most likely to regress silently: Linux mount
adapter behavior, network service checks, shared gRPC client helpers, CLI
negative paths, and test cleanup reliability.

## Context

The repository already has useful unit tests and validation scripts, including
real daemon, Docker, service-forwarding, UDP/datagram, and Linux FUSE mount
validation. The remaining issue is not raw test count. The issue is that some
high-risk behaviors are covered only indirectly by broad scripts or are not
covered at the focused unit/integration boundary where regressions are cheapest
to diagnose.

The review found these real gaps:

- `operon-mount` has only edge coverage for inode/path helpers and an exported
  constant. The Linux validation scripts exercise real mounts, but the FUSE
  operation logic itself lacks focused tests.
- `operon-network` unit tests cover mDNS record conversion/removal, but not TCP
  or UDP service check behavior. A TCP success currently reports a UDP-oriented
  reason string.
- `operon-grpc-client` lacks coverage for `connect()`, request metadata without
  context, non-empty stdin chunking, write chunk boundaries, and connect timeout
  semantics.
- `operon-cli` compiled-binary integration tests mostly cover successful
  command paths.
- Several tests manually remove temporary directories instead of using RAII
  cleanup.
- Token generation is asserted in both `private_files` and `onboard` tests even
  though the onboard test should focus on onboard behavior.

## Scope

1. Mount adapter focused coverage.
   - Add unit tests around reusable mount logic before attempting platform
     adapter expansion.
   - Cover `errno_for_error` mappings for representative tonic status codes.
   - Cover `GrpcRemoteFs` pagination/request behavior through a testable client
     boundary or local tonic test server.
   - Cover FUSE helper behavior that can be tested without a live kernel mount:
     lookup path validation, file attribute mapping, inode refresh on stat,
     write-size reporting, and subtree invalidation after delete/rename.
   - Keep the existing Linux FUSE validation scripts as the live mount coverage
     layer for read, write, truncate, rename, delete, permission-denied, audit,
     and unmount behavior.

2. Network service-check coverage.
   - Fix the TCP success reason string so successful TCP checks report TCP
     reachability.
   - Add loopback TCP tests with a local `TcpListener` for reachable and refused
     service checks.
   - Add UDP tests for connect success and obvious failure paths where they are
     stable on CI.
   - Keep mDNS live discovery out of default unit tests unless it can be made
     deterministic without depending on host multicast behavior.

3. Shared gRPC client helper coverage.
   - Add chunk-boundary tests for `chunk_stdin_requests` and
     `chunk_write_requests`, including bodies larger than `STREAM_CHUNK_BYTES`.
   - Add metadata tests for no auth, auth without execution context, run-only
     context, and step-only context.
   - Add a testable connection deadline around `connect()` and the mount remote
     client connection path so unreachable endpoints do not hang indefinitely.
   - Preserve current endpoint URI normalization semantics for `grpc://` and
     `grpcs://`.

4. CLI negative-path integration coverage.
   - Extend `crates/operon-cli/tests/cli_static_integration.rs` with focused
     compiled-binary failures:
     - unknown command exits non-zero and prints clap error text.
     - missing required command arguments exit non-zero.
     - malformed `config.yaml` exits non-zero for commands that must load config.
     - invalid endpoint scheme is rejected by a command path that constructs a
       gRPC client request.
   - Keep these tests short and static; deeper daemon behavior should stay in
     validation scripts.

5. Test cleanup reliability.
   - Introduce `tempfile::TempDir` for Rust tests that currently use custom
     `unique_temp_dir()` plus manual `remove_dir_all()`.
   - Start with `operon-store`, `operon-cli` static integration, CLI private
     files, CLI init/config tests, and daemon store/config tests where the
     conversion is behavior-preserving.
   - Avoid mocking real filesystem semantics in filesystem crates; use real
     temporary directories with RAII cleanup.

6. Duplicate coverage cleanup.
   - Keep token generation shape tests in `crates/operon-cli/src/private_files.rs`.
   - Replace the duplicate onboard token-generation test with an onboard-specific
     assertion, such as generated plans referencing token files correctly or
     writing expected private-file inputs through the onboard flow.

7. Coverage documentation refresh.
   - Update `docs/quality/test-coverage-audit.md` so it no longer implies the
     current mount/network/grpc-client coverage is stronger than it is.
   - Record which gaps are closed by this phase and which remain broad
     validation-script coverage only.

## Non-Goals

- No numeric line-coverage threshold in this phase.
- No macFUSE or WinFsp implementation.
- No replacement of the Linux FUSE validation scripts with unit tests.
- No broad mocking framework unless a small local trait/test server is needed to
  make a specific gRPC client path deterministic.
- No CLI UX redesign; negative-path tests should preserve current public error
  semantics unless they expose an actual bug.

## Acceptance Criteria

- `operon-network` has deterministic TCP and UDP service-check coverage, and TCP
  success no longer reports a UDP reason.
- `operon-grpc-client` has chunk-boundary, metadata, and connection-deadline
  coverage.
- `operon-mount` has focused tests for mount error mapping and testable FUSE or
  remote-client behavior, while live kernel mount behavior remains validated by
  Linux scripts.
- CLI compiled-binary integration tests include representative failure paths.
- The first RAII cleanup pass removes manual temporary-directory cleanup from
  the targeted tests without reducing real filesystem coverage.
- The duplicate token-generation assertion is replaced with onboard-specific
  coverage.
- `docs/quality/test-coverage-audit.md` and `docs/plan/development-phases.md`
  reflect the updated test-hardening state.
- Validation includes `cargo test --workspace --locked`, the relevant focused
  crate tests, and the consolidated validation group that owns any new
  validation script added for this phase.

## Completed

- Added deterministic `operon-network` TCP/UDP service-check tests and changed
  successful TCP checks to report `tcp service reachable`.
- Added shared `operon-grpc-client` connection deadline handling through
  `connect_channel`, kept `connect()` as the default client constructor, and
  wired the Linux mount remote client through the same deadline path.
- Added `operon-grpc-client` metadata and chunk-boundary tests for write and
  stdin streams larger than `STREAM_CHUNK_BYTES`.
- Added `operon-mount` focused tests for tonic-status errno mapping, FUSE child
  lookup path validation, remote stat caching, and directory attribute mapping.
- Added compiled-binary CLI negative-path tests for unknown commands, missing
  arguments, malformed config files, and invalid endpoint schemes.
- Added `tempfile::TempDir` test cleanup for the targeted CLI, store, and daemon
  tests that previously used custom temporary paths and manual `remove_dir_all`.
- Replaced duplicate onboard token-generation coverage with an onboard-specific
  assertion that daemon onboarding writes a private token file and references it
  from generated config.
- Refreshed `docs/quality/test-coverage-audit.md`.
- Added `scripts/verify-v0.13.6-test-hardening.sh` and wired it into the
  consolidated validation runner.

## Remaining

- No v0.13.6 test-hardening work remains.
- Linux FUSE live mount behavior remains covered by Linux validation scripts
  rather than unit tests.
- Numeric line-coverage thresholds remain a future policy choice.
