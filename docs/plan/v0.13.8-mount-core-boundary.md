# v0.13.8 Mount Core Boundary

Status: Completed.

## Goal

Extract the platform-neutral mount adapter boundary before attempting macFUSE
FSKit or WinFsp native implementation.

## Context

v0.13.7 concluded that supported live mounts should remain Linux-only before
v1.0, and that non-Linux adapter work should not duplicate policy, path,
version, error, and cancellation behavior.

The Linux adapter already has useful modules in
[`operon-mount`](../../crates/operon-mount), but it previously kept the crate
root Linux-gated and defined the remote filesystem trait beside the gRPC client.
A future macFUSE FSKit or WinFsp adapter needs a shared mount-core contract that
can be validated without a live kernel mount.

## Scope

1. Shared mount-core boundary.
   - Isolate platform-neutral `RemoteFs` operation mapping.
   - Keep path validation, stat/list/read/read-range/write-range/truncate/mkdir
     /delete/rename behavior reusable.
   - Keep version/precondition propagation reusable.
   - Keep common error vocabulary and cancellation expectations reusable.

2. Linux adapter preservation.
   - Keep current Linux FUSE behavior unchanged.
   - Keep FUSE callback types, inode table details, and live mount session
     handling in the Linux adapter.
   - Preserve current Linux validation scripts.

3. Validation.
   - Add mount-core unit tests that do not require a live kernel mount.
   - Keep Linux live mount scripts as the integration gate.
   - Ensure docs keep macFUSE and WinFsp as deferred after this phase.

## Non-Goals

- No macFUSE implementation.
- No WinFsp implementation.
- No file sync, offline cache, Finder/Explorer shell polish, or global
  workspace replication.
- No change to the Core FS Protocol unless a concrete protocol gap is proven.

## Acceptance Criteria

- Platform-neutral mount behavior is testable without Linux FUSE.
- Linux FUSE remains supported and behavior-preserving.
- macFUSE and WinFsp adapter implementation remain planned separately.
- `docs/plan/development-phases.md` records what changed and what remains.

## Completed

- Added [`mount_core.rs`](../../crates/operon-mount/src/mount_core.rs) as the
  platform-neutral boundary for `RemoteFs` plus remote path normalization,
  child-name validation, and child path joining.
- Kept [`remote_client.rs`](../../crates/operon-mount/src/remote_client.rs) as
  the gRPC-backed `RemoteFs` implementation and made it depend on
  `mount_core::RemoteFs`.
- Removed the crate-root Linux gate from
  [`operon-mount`](../../crates/operon-mount/src/lib.rs) while keeping Linux
  FUSE adapter modules, inode table, and session management behind
  `#[cfg(target_os = "linux")]`.
- Preserved the current Linux FUSE adapter behavior and kept macFUSE/WinFsp
  implementation deferred.
- Added focused mount-core unit coverage and v0.13.8 validation.
- Aligned the public release line to `0.13.8` / `v0.13.8` across Rust crate
  versions, the TypeScript SDK package version, `PROTOCOL_VERSION`, CLI version
  tests, and validation scripts before publication.
- Published `v0.13.8` from the `main` commit
  `71956cfdde79fb5ba1c9497bd9c19f7a19664762`.
