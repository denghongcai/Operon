# v0.18.7 musl / Alpine Distribution Decision

Status: Completed.

## Goal

Make the Linux musl / Alpine distribution boundary explicit with current
release evidence, then decide whether Operon should keep publishing only
glibc-based Linux archives or plan a separate musl/static artifact line.

## Context

The current public Linux release archives are GNU/glibc binaries. v0.18.5
added real install smoke on `ubuntu:20.04` for the documented glibc 2.31
baseline and `debian:12` for a current stable distribution path. It also
documented Alpine and other musl-based systems as outside the phase because no
musl/static public artifacts exist.

That boundary should not stay implicit. Users on Alpine should see a clear
support decision, and maintainers should know whether future release work
should produce musl/static archives or keep the smaller glibc-only matrix.

## Scope

1. Evidence collection.
   - Add a focused assessment path, tentatively
     `scripts/assess-musl-alpine-distribution.sh`, that runs the current
     public Linux release archive in an Alpine container and records the
     expected unsupported behavior for a glibc-linked binary.
   - Capture the glibc archive baseline from the existing
     `scripts/verify-release-linux-install-containers.sh` checks so the
     decision compares Alpine against the already-supported
     `ubuntu:20.04` / `debian:12` path.
   - Inspect current Rust target feasibility for `x86_64-unknown-linux-musl`
     at the CLI/daemon boundary without committing to shipping a new artifact
     in this phase.

2. Decision record.
   - Add a durable decision document, tentatively
     `docs/decisions/musl-alpine-distribution.md`, that compares:
     glibc-only archives, musl/static Linux archives, Alpine package/container
     distribution, and deferring Alpine support until there is user demand or a
     release blocker.
   - Record operational tradeoffs: binary size, native dependency risk, mount
     runtime behavior, CI cost, support matrix complexity, and user-facing
     installation clarity.
   - Choose one current policy: keep glibc-only and document Alpine as
     unsupported, or create a follow-up implementation phase for musl/static
     artifacts.

3. Documentation alignment.
   - Update `docs/quality/release-install-usability.md`, README, and release
     troubleshooting guidance so Alpine users get a direct support answer
     rather than a generic binary failure.
   - Update `docs/plan/development-phases.md` and `AGENTS.md` with the planned
     and completed decision state.
   - If the decision is to add musl/static artifacts later, add a follow-up
     candidate phase with concrete artifact, CI, docs, and release verification
     scope instead of hiding it as a vague backlog item.

4. Focused validation.
   - Add `scripts/verify-v0.18.7-musl-alpine-distribution-decision.sh` to
     validate that the decision record, release-install docs, README, AGENTS,
     and phase tracker agree on the musl/Alpine support boundary.
   - Wire the validation through the existing consolidated `core` validation
     group if the phase changes release documentation or validation scripts.

## Non-Goals

- No musl/static public release artifact unless this decision explicitly
  creates a follow-up implementation phase.
- No Alpine package, APK repository, Docker image publication, Homebrew, apt,
  yum, winget, Chocolatey, MSI, PKG, DEB, or RPM automation.
- No code signing, notarization, installer UI, or automatic update mechanism.
- No change to the existing glibc baseline release verification unless evidence
  shows the current baseline is wrong.
- No change to Operon's network boundary or mount runtime support policy.

## Acceptance

- The repo contains current evidence for what happens when the public glibc
  Linux archive is used on Alpine or another musl-based container.
- The support policy for musl/Alpine is explicit in the decision record and
  user-facing release/install docs.
- The phase either closes with a glibc-only decision and clear unsupported
  Alpine guidance, or creates a concrete follow-up implementation phase for
  musl/static artifacts.
- Existing glibc release install usability checks remain the supported Linux
  release gate unless the decision changes the artifact strategy.
- AGENTS guidance and the phase tracker point future agents to the chosen
  musl/Alpine policy.
