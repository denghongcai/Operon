# v0.6.2 CLI FS Cleanup Acceptance

Status: Completed.

v0.6.2 is scoped to aligning the human CLI with the filesystem mutation
operations added for v0.6.1.

It includes:

- `operon fs mkdir <node:/path>`.
- `operon fs rm <node:/path>`.
- `operon fs rename <node:/from> <node:/to>`.
- `operon fs truncate <node:/path> --size <bytes>`.
- README and protocol examples for the new CLI commands.
- validation for successful mutation commands and policy-denied mutation
  commands.

It deliberately does not include a new mount mode, a new daemon capability, or a
new network provider. `WriteFileRange` remains a low-level Core FS Protocol
operation for mount adapters and SDK/direct protocol clients unless a concrete
CLI use case appears.

## Contract

The CLI should map directly to the Core FS Protocol:

- `fs mkdir` maps to `MkdirFs`.
- `fs rm` maps to `DeleteFs`.
- `fs rename` maps to `RenameFs`.
- `fs truncate` maps to `TruncateFs`.

The daemon remains authoritative for path containment, policy, and audit. CLI
commands should report daemon errors without trying to duplicate daemon-side
authorization logic.

## Concurrency And Conflict Semantics

The v0.6.2 CLI uses the same filesystem concurrency contract as v0.6.1 mount:

- CLI mutation commands do not pass expected file versions or etags.
- CLI mutation commands do not acquire locks or leases.
- concurrent CLI, mount, SDK, or direct protocol writes to the same path are not
  coordinated by Operon.
- daemon audit records what happened, but audit is not a conflict detector.

Users who need deterministic mutation ordering should serialize operations at
the workflow, CLI, or agent layer until a later optimistic concurrency or lease
phase exists.

## Validation

The canonical v0.6.2 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
scripts/verify-v0.6.2-cli-fs-cleanup.sh
```

The CLI cleanup validation must prove:

- mkdir creates a remote directory.
- truncate creates or resizes a remote file deterministically.
- rename moves a remote path.
- rm deletes a remote file or empty directory.
- denied mkdir/truncate/rm/rename operations are audited by the remote node.

## Known Limits

- `WriteFileRange` is not exposed as a first-class human CLI command in v0.6.2.
- Recursive delete is not part of v0.6.2 unless explicitly added to the daemon
  protocol later.
- Version preconditions, locks, leases, and multi-writer conflict detection are
  not part of v0.6.2.
