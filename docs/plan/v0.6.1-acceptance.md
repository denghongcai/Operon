# v0.6.1 Acceptance

v0.6.1 is scoped to Linux write mount support.

It includes:

- Linux-only write-through FUSE mount behavior.
- write operations over the existing Operon fs capability boundary.
- daemon-owned policy enforcement for write, delete, and rename operations.
- audit records for successful and denied write mount operations.
- validation for read-after-write and cleanup after write failures.

It deliberately does not include WinFsp, macOS mount support, offline sync,
distributed cache invalidation, multi-writer conflict resolution, or a separate
FUSE-specific capability model.

## Contract

v0.6.1 uses Linux-only, single-writer, write-through semantics:

- `create` creates an empty remote file through the Core FS Protocol.
- `write` maps each FUSE write to `WriteFileRange(path, offset, data)`.
- `flush` and `fsync` are acknowledgement points only; writes are already sent
  through to the daemon.
- `truncate` maps to `TruncateFs(path, size)`.
- `mkdir` maps to `MkdirFs(path)`.
- `unlink` and `rmdir` map to `DeleteFs(path)`.
- `rename` maps to `RenameFs(from_path, to_path)` and requires delete
  permission on the source plus write permission on the target.
- FUSE errors are derived from gRPC status codes. Policy denial maps to access
  denied, missing paths map to not found, and unsupported operations such as
  symlink or hardlink return unimplemented.
- The mount adapter keeps only transient inode state. Policy, audit, path
  containment, and actual filesystem mutation remain owned by `operond`.

## Concurrency And Conflict Semantics

v0.6.1 does not provide file conflict detection or conflict resolution.

The current contract is intentionally simple:

- one writer per mounted workspace is the expected operating model.
- reads are live reads against the remote daemon, not snapshot reads.
- writes are write-through RPCs; there is no client-side merge queue.
- concurrent writers to the same path are not coordinated by Operon.
- there is no file version, etag, compare-and-swap precondition, lock, lease, or
  distributed transaction in this phase.
- if two clients write the same file concurrently, the visible result is
  determined by the remote filesystem and RPC arrival order.
- if a client reads while another client writes, the read may observe whatever
  content the remote filesystem exposes at that moment.

This is acceptable for single-operator and controlled agent workflows. It is not
a collaborative editing, offline sync, or multi-writer filesystem contract.

## Validation

The canonical v0.6.1 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
scripts/verify-v0.6.1-linux-write-mount.sh
```

The Linux write mount validation must prove:

- creating a file through the mount writes to the remote node.
- read-after-write returns the written content through both FUSE and `operon fs`.
- truncating or overwriting a file is deterministic.
- delete and rename behavior matches the documented contract.
- denied write/delete/rename operations are audited by the remote node.
- unmount and signal cleanup remain reliable after write failures.

## Completed Implementation

Completed v0.6.1 implementation:

- extended [`proto/operon/runtime.proto`](../../proto/operon/runtime.proto) with write-range, truncate, mkdir,
  delete, and rename filesystem RPCs.
- implemented daemon handlers for the new fs mutation RPCs with existing
  workspace path containment, policy checks, and audit events.
- extended `RemoteFs` and `GrpcRemoteFs` with write-capable filesystem
  operations.
- upgraded the Linux FUSE adapter from read-only to write-through.
- added [`scripts/verify-v0.6.1-linux-write-mount.sh`](../../scripts/verify-v0.6.1-linux-write-mount.sh).
- updated CI with a v0.6.1 Linux write mount validation job.

## Known Limits

- v0.6.1 targets Linux only.
- v0.6.1 should use single-writer, write-through semantics.
- Offline sync, conflict resolution, and cross-platform mount support remain
  later work.
- Multi-writer conflict detection, file leases, optimistic version checks, and
  snapshot reads remain later work.
