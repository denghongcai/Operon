# v0.8.7 Filesystem Service Reuse Cleanup

Status: Completed.

Goal: reduce repeated filesystem authorization, path resolution, and audit
denial handling in the daemon filesystem service without changing runtime
behavior or protocol shape.

Review finding:

- `crates/operond/src/fs_service.rs` repeated the same policy authorization,
  workspace path resolution, failed-audit recording, and `tonic::Status`
  conversion pattern across `stat`, `list`, `read_range`, `write_range`,
  `truncate`, `mkdir`, `delete`, `rename`, and `copy`.
- The repetition made later filesystem operations more likely to drift in audit
  behavior or permission mapping.

Completed:

- Added focused helpers for filesystem authorization and read/write/create/leaf
  workspace path resolution.
- Kept each operation's existing permission action, audit action, audit
  resource, and success audit behavior unchanged.
- Added `scripts/verify-v0.8.7-fs-service-reuse-cleanup.sh` to validate the
  helper boundary and run the daemon tests.

Validation:

```bash
scripts/verify-v0.8.7-fs-service-reuse-cleanup.sh
cargo fmt --check
cargo clippy --workspace --locked -- -D warnings
cargo test --workspace --locked
```

Remaining:

- No v0.8.7 work remains.
- Moving the full tonic `GrpcRuntime` trait implementation out of
  `operond/src/main.rs` remains a future maintainability candidate if runtime
  method routing grows again.
