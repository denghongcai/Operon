# v0.8.4 Runtime and CLI Modularization

Status: Completed.

## Goal

Reduce the largest maintenance hotspots without changing runtime behavior.

This phase is a behavior-preserving refactor. It should make future protocol,
service, provider, and CLI work cheaper by separating domain responsibilities
that are currently concentrated in large entrypoint files.

## Scope

### Daemon Split

Split [`crates/operond/src/main.rs`](../../crates/operond/src/main.rs) so `main.rs` keeps only startup wiring and
top-level command dispatch. Move runtime responsibilities into focused modules:

- `server` or `runtime_service`: gRPC service implementation and RPC routing.
- `state`: daemon shared state and state initialization.
- `fs_service`: filesystem RPC handlers.
- `job_runtime`: job lifecycle, process spawning, termination, stdin, and status
  events.
- `job_logs`: append-only log buffers, streaming snapshots, truncation metadata,
  and retention.
- `service_forward`: TCP service tunnel handling.
- `service_datagram`: UDP/datagram forwarding.
- `audit`: audit event construction, context propagation, and append behavior.
- `pagination`: shared page-token helpers.

### CLI Split

Split [`crates/operon-cli/src/main.rs`](../../crates/operon-cli/src/main.rs) so `main.rs` owns only clap model
construction and high-level dispatch. Move command and rendering logic into
focused modules:

- [`commands/fs.rs`](../../crates/operon-cli/src/commands/fs.rs)
- [`commands/job.rs`](../../crates/operon-cli/src/commands/job.rs)
- [`commands/service.rs`](../../crates/operon-cli/src/commands/service.rs)
- [`commands/audit.rs`](../../crates/operon-cli/src/commands/audit.rs)
- [`commands/config.rs`](../../crates/operon-cli/src/commands/config.rs)
- [`commands/node.rs`](../../crates/operon-cli/src/commands/node.rs)
- [`commands/trace.rs`](../../crates/operon-cli/src/commands/trace.rs)
- [`commands/init.rs`](../../crates/operon-cli/src/commands/init.rs)
- [`commands/mount.rs`](../../crates/operon-cli/src/commands/mount.rs)
- [`output.rs`](../../crates/operon-cli/src/output.rs)
- [`target.rs`](../../crates/operon-cli/src/target.rs)

Where practical, command handlers should return structured values and let a
shared renderer handle text, JSON, and quiet output. If a full renderer
inversion would expand the phase too much, document the remaining command
families that still render directly.

## Non-Goals

- No runtime protocol changes.
- No new capability behavior.
- No endpoint discovery UX implementation.
- No broad public CLI redesign.
- No change to existing JSON/quiet/failure semantics except where tests reveal
  accidental drift during the split.

## Done When

- [`operond/src/main.rs`](../../crates/operond/src/main.rs) no longer owns fs, job, service-forwarding, audit, and
  pagination implementation details directly.
- [`operon-cli/src/main.rs`](../../crates/operon-cli/src/main.rs) no longer owns command handlers and renderers for
  every command family directly.
- The split is behavior-preserving and covered by existing validation.
- Any intentionally deferred extraction is documented with the reason and the
  remaining owner module.

Deferred extraction:

- Job runtime, service forwarding, audit helpers, and non-fs CLI command
  families still have substantial logic in the entrypoint files. They remain
  behaviorally unchanged in this phase to avoid mixing a broad refactor with the
  `ReadFileRange` protocol change. They should be split in follow-up
  maintainability phases before large job/service feature work.
