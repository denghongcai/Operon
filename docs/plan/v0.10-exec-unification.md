# v0.10 Execution Capability Unification

Status: Completed.

Goal: remove the historical `job` concept from Operon's active user and agent
surface, and make remote execution read as one coherent `exec` capability.

## Scope

1. Active runtime protocol.
   - Replace `RunJob`, `GetJob`, `ListJobs`, `WatchJob`, `ListJobLogs`,
     `StreamJobLogs`, `WriteJobStdin`, `CloseJobStdin`, and `CancelJob` with
     the matching `Exec` RPCs.
   - Replace active protobuf messages and enums with `Exec*` names.
   - Replace `CAPABILITY_KIND_JOB` with `CAPABILITY_KIND_EXEC`.

2. Runtime, policy, and audit vocabulary.
   - Replace `policy.job` with `policy.exec`.
   - Replace `job:default` with `exec:default`.
   - Replace job deny reason codes with `exec-cwd-denied` and
     `exec-timeout-exceeded`.
   - Keep shell-string and shell-free `argv[]` modes under `exec.run`.
   - Keep execution records, logs, status, cancellation, timeout, stdin piping,
     environment policy, secrets, persistence, and audit behavior.

3. CLI and SDK surface.
   - Replace `operon job ...` with `operon exec ...`.
   - Do not keep `operon job` as a compatibility alias.
   - Replace TypeScript SDK job helpers with exec helpers such as `runExec`,
     `getExec`, `listExecs`, `streamExecLogs`, and `writeExecStdin`.

4. Graph, examples, docs, and skills.
   - Replace active graph action names from `job.run` to `exec.run`.
   - Replace active examples and validation scripts with exec terminology.
   - Update README, `PROTOCOL.md`, runtime API docs, development docs,
     repo-local skills, and AGENTS guidance.

5. Future interactive mode.
   - Document `exec.session` as the future PTY/TTY-backed interactive execution
     mode.
   - Do not add an unimplemented `exec.session` RPC in v0.10.

## Non-Goals

- No conservative `job` compatibility alias.
- No PTY/TTY session implementation in this phase.
- No change to Operon's network boundary.
- No new scheduling, queueing, orchestration, or container runtime behavior.

## Done When

- Active CLI help exposes `operon exec ...` commands and rejects `operon job`.
- Active gRPC protocol exposes `Exec*` RPCs, messages, and enums with no active
  `Job*` protocol names.
- Rust daemon, CLI, core DTOs, policy, store, and process helpers use exec
  vocabulary.
- TypeScript SDK exposes exec helpers and generated bindings.
- Active docs, examples, repo-local skills, and validation scripts use
  `exec.run`, `exec:default`, and `policy.exec`.
- `scripts/verify-v0.10-exec-unification.sh` is added and wired into CI and
  development validation.
- Existing runtime behavior remains covered by unit tests and integration
  validation under the new exec vocabulary.

## Completed

- Replaced the active gRPC runtime execution surface with `Exec*` RPCs,
  messages, enums, and `CAPABILITY_KIND_EXEC`.
- Replaced daemon, core DTO, process, store, CLI, graph, policy, audit, and
  TypeScript SDK vocabulary with `exec`.
- Replaced active CLI commands with `operon exec ...` and intentionally removed
  `operon job` as a supported command.
- Updated active examples, README, `PROTOCOL.md`, runtime API docs,
  DEVELOPMENT.md, AGENTS.md, repo-local skills, and validation scripts.
- Added [`scripts/verify-v0.10-exec-unification.sh`](../../scripts/verify-v0.10-exec-unification.sh)
  and wired it into CI and Full Validation.
- Bumped Rust crate versions, the TypeScript SDK package version, and
  `PROTOCOL_VERSION` to `v0.10.0` / `0.10.0`.

## Remaining

- Nothing remains in v0.10.
