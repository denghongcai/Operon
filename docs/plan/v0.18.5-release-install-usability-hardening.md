# v0.18.5 Release / Install Usability Hardening

Status: Completed.

## Goal

Harden the path from a published GitHub Release asset to a working local
Operon installation, so release completion is validated from the user's
downloaded artifact rather than only from the source tree or draft release
workflow.

## Context

The current release process already builds archive artifacts, runs archive
smoke tests before upload, publishes checksums, verifies public release assets,
and validates the README Quickstart after publication. The remaining gap is
post-download usability: a clean machine or container should be able to fetch a
public asset, install the binaries into a temporary or user-level location, run
the first useful commands, and surface platform prerequisite failures clearly.

This phase should make that path repeatable without expanding Operon into an
OS package manager, installer product, code-signing pipeline, or network
provisioning layer.

## Scope

1. Public asset install smoke.
   - Add a release-install smoke path that downloads a published release tag,
     verifies checksums, extracts the current platform archive, installs
     `operon` and `operond` into an isolated test prefix, and proves the
     installed binaries are used from `PATH`.
   - Exercise `operon --version`, `operond --version`, top-level help,
     `operon init config`, `operon doctor`, a foreground local daemon, node
     ping, capability listing, and one minimal filesystem read/write loop where
     platform permissions allow it.
   - Keep service-manager installation and live mount runtime checks as
     platform-specific optional probes unless the platform can run them safely
     in GitHub Actions.

2. Linux distribution and glibc compatibility.
   - Add container-based Linux install smoke for the documented minimum glibc
     baseline and at least one current stable distribution image.
   - Keep symbol-version checks in the release workflow, but add a real
     extracted-binary smoke so compatibility failures are caught as runtime
     errors, not only as `objdump` output.
   - Document unsupported musl/Alpine behavior as a clear limitation unless a
     later phase defines musl/static release artifacts.

3. macOS and Windows post-download checks.
   - Reuse the public release asset verifier where possible, but add
     install-location and `PATH` checks that mimic how README users run the
     downloaded binaries.
   - Keep WinFsp and FUSE-T live mount prerequisites in the existing live mount
     release gates; this phase only needs to ensure post-download mount-runtime
     diagnostics are visible and actionable.

4. README and troubleshooting alignment.
   - Update README Quickstart and release troubleshooting docs so install
     paths, archive names, glibc requirements, mount runtime prerequisites, and
     expected verification commands match the automated install smoke.
   - Make the documented post-release maintainer checklist point to the exact
     local command or manual workflow that proves install usability for a tag.

5. Validation and workflow wiring.
   - Add a focused validation script for release-install usability wiring and
     dry-run behavior.
   - Add or extend a manual GitHub Actions workflow that can verify a published
     tag after release without requiring a source checkout to stand in for the
     installed product.
   - Wire the narrow local checks through the existing consolidated validation
     groups instead of adding version-specific CI jobs.

## Non-Goals

- No npm, crates.io, Homebrew, apt, yum, winget, Chocolatey, or MSI/PKG/DEB/RPM
  publishing automation.
- No code signing, notarization, or installer UI.
- No automatic update mechanism.
- No change to release versioning, public artifact naming, or the existing
  draft release state machine unless validation exposes an actual mismatch.
- No new networking layer, VPN behavior, relay service, or reachability
  automation.
- No new musl/static Linux artifact unless a separate phase defines that
  distribution target.

## Acceptance

- A maintainer can run one documented command or manual workflow against a
  published release tag and verify that downloaded artifacts install into an
  isolated location and run the first useful Operon workflow from `PATH`.
- Linux install smoke covers the documented glibc baseline with a real
  extracted binary, not only symbol inspection.
- macOS and Windows install smoke prove downloaded binaries launch and expose
  actionable doctor output for mount runtime prerequisites.
- README Quickstart, release troubleshooting docs, AGENTS guidance, and
  `docs/plan/development-phases.md` agree on the post-release install
  verification path.
- The release-install usability validation can run locally in dry-run mode
  without publishing a new release.

## Progress

- Added `scripts/verify-release-install-usability.sh` to download the current
  platform's public release archive and `SHA256SUMS`, verify the archive
  checksum, install `operon` and `operond` into an isolated prefix, prove
  `PATH` resolves the installed binaries from that prefix, run version/help and
  `operon doctor --mount-runtime`, create starter config, start a local
  foreground daemon, and exercise node, capability, filesystem, doctor, and
  audit commands from the installed product.
- Added `scripts/verify-release-linux-install-containers.sh` to run the
  release-install smoke inside `ubuntu:20.04` for the documented glibc 2.31
  baseline and `debian:12` for a current stable distribution path.
- Added the manual `Verify Release Install Usability` workflow for Ubuntu,
  macOS, Windows, and Linux container install usability checks against a
  published tag.
- Added `docs/quality/release-install-usability.md` with the post-download
  verification path, dry-run commands, glibc/musl boundary, and failure triage.
- Updated README install guidance so macOS users keep bundled
  `libfuse-t.dylib` next to the installed binaries, and documented local
  dry-run commands for install-usability workflow wiring.
- Added `scripts/verify-v0.18.5-release-install-usability-hardening.sh` and
  wired it through the consolidated `core` validation group.
