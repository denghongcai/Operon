# v0.13.1 Windows PTY Validation

Status: Planned.

## Goal

Replace the current Windows PTY validation deferral with an explicit,
runner-safe decision: supported, degraded, or intentionally unsupported for the
current release line.

## Context

The Windows CI platform smoke previously hung in
`exec_session_portable_pty_smoke_outputs_and_exits`. CI now skips that smoke on
Windows and `operon doctor` reports `portable-pty-validation-deferred` there.
That is truthful, but it should not remain ambiguous.

## Scope

1. Failure-mode investigation.
   - Reproduce the Windows PTY smoke behavior in GitHub Actions logs or a
     dedicated short-timeout validation job.
   - Distinguish portable-pty backend limitations from test harness issues.
   - Add hard timeouts around any Windows PTY probe so CI cannot hang.

2. Capability decision.
   - Decide whether Windows `exec session` is supported, degraded, or
     unsupported for this release line.
   - Keep Unix/macOS PTY behavior unchanged.
   - Preserve non-interactive Windows exec and Job Object cancellation
     guarantees.

3. UX and docs.
   - Update `operon doctor`, README, `PROTOCOL.md`, and runtime API docs with
     the exact Windows PTY status.
   - If Windows PTY is unsupported, make CLI/server errors explicit instead of
     letting sessions hang.

4. Validation.
   - Add a Windows-safe CI validation path for the chosen status.
   - Keep the existing Unix-like `portable-pty` smoke coverage.

## Non-Goals

- No replacement PTY dependency beyond `portable-pty`.
- No remote desktop, terminal multiplexer, or shell UX redesign.
- No mount adapter work.

## Acceptance Criteria

- Windows PTY behavior no longer has an ambiguous validation state.
- CI cannot hang on the Windows PTY probe.
- `operon doctor` and docs report the implemented Windows status.
- Existing Unix-like PTY session tests continue to pass.
