# v0.7 Acceptance

v0.7 is scoped to service metadata, service health checks, and explicit local
port forwarding for policy-allowed services.

The previous CLI TUI console milestone is cancelled. Operon remains CLI, SDK,
and protocol first.

It includes:

- `ListServices` service metadata.
- `CheckService` TCP health checks.
- `OpenServiceTunnel` bidirectional gRPC streams.
- `operon service forward <node-id> <service-id> --listen <addr>`.
- TypeScript SDK `openServiceTunnel` over `nice-grpc`.
- audit events for service forwarding attempts.
- documentation for direct protocol clients that do not use an SDK.

It deliberately does not include VPN behavior, NAT traversal, relay networking,
mesh IP assignment, global routing, unmanaged port exposure, browser UI, or a
CLI TUI console.

## Validation

The canonical v0.7 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm test
scripts/verify-v0.7-service-forwarding.sh
```

The service forwarding validation must prove:

- the daemon can list and health-check a policy-configured TCP service.
- the CLI can bind a local listener and forward traffic to that service.
- forwarded bytes are transported over the existing gRPC node connection.
- service forwarding records audit events.
- no HTTP runtime facade is reintroduced.

## Known Limits

- Forwarding is local and explicit. A client must run `operon service forward`
  or implement `OpenServiceTunnel` directly.
- Operon does not discover, allocate, or route service ports globally.
- Policy controls which daemon-local services can be inspected or forwarded;
  network reachability between Operon nodes is still provided by the existing
  private network layer.
