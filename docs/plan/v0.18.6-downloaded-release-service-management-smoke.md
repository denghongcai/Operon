# v0.18.6 Downloaded Release Service-Management Smoke

Status: Completed.

## Goal

Extend post-download release validation so a published GitHub Release archive
proves that `operond service install/start/status/stop/uninstall` works from
the installed release binary path, not only from a source-tree build.

## Context

v0.13.5 added platform-native daemon service management and validated the
source-built command surface, rendered systemd and launchd definitions, and
Windows Service registration command shape. v0.18.5 added downloaded-release
install usability validation, but it stops at a foreground local daemon smoke.

The remaining gap is the user path after installation: a downloaded `operond`
binary should install a platform supervisor entry that points back to the
installed binary and explicit config file, while surfacing clear status and
cleanup behavior. CI should validate this without requiring persistent real
systemd, launchd, or elevated Windows Service registration.

## Scope

1. Downloaded service-management smoke script.
   - Add a focused script, tentatively
     `scripts/verify-release-service-management-smoke.sh`, that downloads a
     public release tag, verifies checksums, extracts the current platform
     archive, installs `operon` and `operond` into an isolated prefix, and
     proves `PATH` resolves the installed binaries from that prefix.
   - Reuse or extract shared release-install helper logic from
     `scripts/verify-release-install-usability.sh` instead of copying the full
     download, checksum, archive, install-prefix, and `PATH` setup flow into a
     second large script.
   - Generate an isolated config with `operon onboard` so service install uses
     a real config path and token file under a temporary `HOME`.

2. Safe platform supervisor simulation.
   - On Linux, put a fake `systemctl` earlier in `PATH`, run the installed
     `operond service install/start/status/stop/uninstall`, and assert the
     generated user systemd unit under the isolated `XDG_CONFIG_HOME` invokes
     the installed `operond start --config <path>` binary and config.
   - On macOS, put a fake `launchctl` earlier in `PATH`, run the installed
     service commands, and assert the generated launchd plist under the
     isolated `HOME/Library/LaunchAgents` invokes the installed binary and
     config.
   - On Windows, put a fake `sc.exe` earlier in `PATH`, run the installed
     service commands where supported, and assert the service create arguments
     point at the installed `operond.exe service run --config <path>` command.
   - Keep fake supervisor logs as workflow evidence so failures show whether
     the installed binary, generated service definition, or command sequence
     drifted.

3. Workflow integration.
   - Extend `.github/workflows/verify-release-install-usability.yml` with a
     downloaded service-management smoke step or job for Ubuntu, macOS, and
     Windows.
   - Keep the existing foreground local daemon smoke and Linux container glibc
     smoke intact; this phase adds service-management coverage rather than
     replacing the v0.18.5 checks.
   - Keep real persistent supervisor start/stop optional and manual when a
     runner cannot safely provide user systemd, launchd, or elevated Windows
     Service registration.

4. Documentation and release checklist.
   - Update `docs/quality/release-install-usability.md` to document the new
     downloaded service-management smoke and explain that CI uses fake
     supervisor commands to validate generated service definitions safely.
   - Update README and DEVELOPMENT release-validation guidance if command names
     or maintainer workflow steps change.
   - Update `AGENTS.md` and `docs/plan/development-phases.md` with the planned
     and completed phase state.

5. Focused validation.
   - Add `scripts/verify-v0.18.6-downloaded-release-service-management-smoke.sh`
     to check the phase doc, workflow wiring, service-management smoke script,
     dry-run behavior, and docs alignment.
   - Wire the focused validation through the existing consolidated `core`
     validation group in `scripts/ci/run-validations.sh`.

## Non-Goals

- No `operond start --background` or self-daemonizing process mode.
- No replacement for systemd, launchd, Windows Service Control Manager, Docker,
  Kubernetes, or any other supervisor.
- No requirement that GitHub-hosted CI create a real persistent user service.
- No elevated Windows service-registration requirement in automated CI.
- No installer UI, code signing, notarization, or package-manager publishing.
- No change to release archive names or public release version policy.

## Acceptance

- A maintainer can run a documented command or manual workflow against a public
  tag and prove service-management commands execute from the downloaded
  release binary, not from the source checkout.
- Linux, macOS, and Windows checks prove generated supervisor definitions or
  registration arguments point at the isolated installed `operond` path and
  explicit config path.
- The smoke runs without leaving real services installed on CI runners.
- Existing v0.18.5 release install usability and Linux glibc container checks
  continue to pass.
- README, DEVELOPMENT, release troubleshooting docs, AGENTS guidance, and the
  phase tracker agree on the downloaded service-management validation path.
