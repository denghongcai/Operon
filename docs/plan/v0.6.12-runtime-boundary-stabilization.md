# v0.6.12 Runtime Boundary Stabilization

v0.6.12 stabilizes the long-term runtime boundaries before v0.7 TUI work.

This phase should avoid short-lived compatibility patches. The goal is to make
the protocol, store, job log stream, and mount adapter boundaries explicit
enough that CLI, SDK, and future TUI code can build on them without duplicating
daemon internals.

## Scope

- Replace raw `StreamJobLogs` chunks with a streaming envelope that can carry
  snapshots, log entries, and terminal completion metadata.
- Keep `ListJobLogs` as the paginated/snapshot-style query API, not as a fake
  streaming result.
- Promote `operon-store` from a best-effort helper to an explicit append-only
  event writer boundary with visible fsync policy.
- Make daemon persistence failures visible at the daemon boundary instead of
  being swallowed inside `operon-store`.
- Reduce scattered background lock handling by routing job/log/audit mutations
  through explicit runtime helper boundaries.
- Make the mount implementation an explicit Linux FUSE adapter boundary, with
  non-Linux platforms excluded at the mount crate dependency boundary.

## Non-Goals

- No HTTP runtime facade.
- No new capability surface.
- No macFUSE or WinFsp implementation.
- No full actor rewrite of the daemon in this phase; this phase establishes the
  APIs and ownership points needed for that migration.

## Acceptance

- `StreamJobLogs` returns a typed envelope rather than bare `JobLog` messages.
- CLI JSON and streaming log paths consume the envelope and preserve truncation
  metadata.
- TS SDK consumes the generated envelope API and exposes real log stream events.
- `operon-store` exposes `StoreWriter` and append failures return `Result`.
- Daemon store append failures are logged at the daemon boundary.
- `operon-mount` only depends on `fuser` on Linux.
- CI runs a v0.6.12 runtime-boundary validation script.
- Full Rust and TypeScript validation passes.
