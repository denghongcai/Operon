# v0.13.3 Config and Onboard Maintainability Cleanup

Status: Planned.

## Goal

Reduce future config UX change cost by splitting the remaining config and
onboard command responsibilities into clearer plan, render, and write
boundaries.

## Scope

1. Config command boundary.
   - Reassess `crates/operon-cli/src/commands/config.rs` responsibility
     clusters.
   - Separate config explain data collection, rendering, and command dispatch
     where behavior can be preserved.
   - Preserve text, JSON, and error contracts.

2. Onboard boundary.
   - Reassess `crates/operon-cli/src/commands/onboard.rs` plan/write/render
     responsibilities.
   - Keep guided onboarding as a wrapper over normal config files and CLI setup
     primitives.
   - Preserve generated config shape and token-file behavior.

3. Validation.
   - Add focused maintainability validation for extracted module boundaries.
   - Run docs/help/skills sync if public help or docs are touched.

## Non-Goals

- No config schema redesign.
- No new onboarding product flow.
- No endpoint/discovery behavior change.
- No mount adapter work.

## Acceptance Criteria

- Config and onboard command behavior remains unchanged.
- New module boundaries make plan/write/render responsibilities explicit.
- Existing CLI JSON/text contracts remain stable.
- CI includes focused validation for the cleanup.
