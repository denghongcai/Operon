# v0.14 Cross-Platform Live Mount

Status: In progress.

## Goal

Make live mount a complete core Operon capability across Linux, macOS, and
Windows instead of treating non-Linux mount support as deferred convenience
work.

## Context

v0.13.7 documented macFUSE and WinFsp implementation strategy and chose a
pre-v1.0 Linux-only live mount boundary. v0.13.8 then extracted the first
platform-neutral `mount_core` boundary. That boundary is useful, but it is not
enough for the product goal: users should be able to expose an Operon
filesystem as a live local mount on all primary release platforms.

This phase supersedes the v0.13.7 Linux-only pre-v1.0 support decision. The
Core FS Protocol remains the authoritative filesystem API, but live mount is now
treated as a core adapter capability that must reach macOS and Windows.

## Scope

1. Shared mount adapter core.
   - Move reusable mount behavior out of the Linux FUSE adapter:
     path normalization, child validation, remote operation mapping, stat/list
     conversion, read-range/write-range behavior, truncation,
     mkdir/delete/rename semantics, error vocabulary, cancellation expectations,
     and common test fixtures.
   - Keep daemon policy, version/precondition checks, audit, and workspace
     containment authoritative on the daemon side. Mount adapters must not
     bypass Core FS Protocol semantics.
   - Keep platform adapters thin: they translate OS filesystem callbacks into
     shared mount-core operations.

2. macOS live mount.
   - Convert the current Linux-only FUSE adapter into a Unix FUSE adapter that
     supports Linux and macOS where the `fuser` crate and macFUSE runtime are
     available.
   - Make `operon mount` available on macOS builds.
   - Add macOS-specific missing-runtime diagnostics for macFUSE installation,
     permission, and backend limitations.
   - Start with the macFUSE path that can reuse the existing FUSE callback
     model; do not build Finder shell polish or File Provider-style sync in
     this phase.

3. Windows live mount.
   - Add a native WinFsp adapter after the shared core and macOS adapter are
     working.
   - Prefer MIT-compatible bindings such as `winfsp_wrs` or direct FFI if the
     binding surface is too small. Do not add GPLv3 `winfsp` to the Apache-2.0
     workspace without an explicit license decision.
   - Make `operon mount` available on Windows builds when WinFsp support is
     compiled in.
   - Add missing-runtime diagnostics for the WinFsp driver/runtime and keep
     Windows filesystem semantics explicit instead of pretending they are POSIX.

4. CLI, doctor, docs, and release alignment.
   - Replace the current non-Linux `operon mount is only supported on Linux`
     error with platform-aware support and diagnostics.
   - Extend `operon doctor` to explain mount adapter status on Linux, macOS,
     and Windows.
   - Update README, PROTOCOL, runtime API docs, AGENTS guidance, and repo-local
     skills when the implemented support boundary changes.
   - Update release notes and validation scripts so public artifacts truthfully
     describe which live mount paths are supported, experimental, or blocked by
     missing platform runtime dependencies.

## Implementation Order

1. Plan and validation scaffolding.
   - Add this phase and wire a static validation script that checks the intended
     platform gates, docs, and CLI support wording.
   - Keep Linux mount validation green before changing adapter boundaries.

2. Shared mount-core expansion.
   - Add mount-core unit tests first for operation mapping, stat conversion,
     path/name validation, remote error classification, and cancellation
     behavior that can be exercised without a live kernel mount.
   - Move reusable logic out of the current Linux FUSE callback module only
     after tests fail against the current boundary.

3. Unix FUSE adapter.
   - Rename or split the Linux-specific adapter into a Unix FUSE adapter where
     practical.
   - Keep Linux behavior unchanged and validated by existing Linux live mount
     scripts.
   - Enable macOS compile/test paths with `fuser` and macFUSE-oriented
     diagnostics.

4. macOS smoke path.
   - Add normal GitHub-hosted macOS compile and unit validation.
   - Add a manual or self-hosted macOS live mount smoke plan if hosted runners
     cannot install or use macFUSE reliably.
   - Smoke cases: mount, list, stat, read, random read, write, truncate, mkdir,
     rename, delete, permission denied, unmount, and daemon disconnect.

5. Windows WinFsp adapter.
   - Add the Windows adapter behind explicit Windows `cfg` gates.
   - Keep WinFsp dependency optional or target-scoped.
   - Add unit tests for UTF-16 path handling, Windows error mapping, read/write
     operation dispatch, delete/rename semantics, missing WinFsp runtime
     diagnostics, and unmount/cancellation behavior.

6. Windows smoke path.
   - Add a Windows CI or temporary Actions smoke that installs WinFsp only after
     the dependency and runner permissions are proven.
   - Smoke cases: mount or drive exposure, list, stat, read, random read,
     write, truncate, mkdir, rename, delete, permission denied, unmount, and
     daemon/service stop.

7. Public release hardening.
   - Align versions, docs, CLI help, `operon doctor`, release workflows, and
     release artifact verification before publishing a release that claims
     cross-platform live mount support.
   - Run README Quickstart and release artifact validation against the public
     release after publication.

## Non-Goals

- No offline file sync engine.
- No global workspace replication.
- No File Provider, Explorer shell extension, Finder badge, preview, or shell
  polish.
- No conflict resolution beyond the existing Core FS Protocol version and
  precondition semantics.
- No Windows ACL editing or alternate data stream support in the first Windows
  adapter.
- No POSIX uid/gid/mode fidelity guarantee on Windows.
- No macFUSE kernel-extension security downgrade as the default documented
  path.
- No transport, VPN, relay, NAT traversal, or network mesh responsibility.

## Acceptance Criteria

- `operon mount` supports Linux, macOS, and Windows release builds with
  platform-aware diagnostics.
- Linux live mount behavior remains unchanged and validated by existing Linux
  live mount scripts.
- macOS has a real live mount adapter path, compile/unit validation in CI, and
  a documented live smoke gate.
- Windows has a real WinFsp native adapter path, compile/unit validation in CI,
  and a documented or automated live smoke gate.
- Missing macFUSE or WinFsp runtime dependencies produce actionable CLI and
  `operon doctor` diagnostics.
- Shared mount-core tests cover platform-neutral path, stat, read/write range,
  truncate, mkdir, delete, rename, error, and cancellation behavior.
- README, PROTOCOL, runtime API docs, AGENTS guidance, repo-local skills, and
  release docs match the implemented support boundary.

- Local validation has passed for `scripts/ci/run-validations.sh core`,

  `scripts/ci/run-validations.sh linux-system`.
- The first remote CI pass found and fixed platform validation wiring issues:
  macOS non-live CLI tests now use the `operon-mount/macos-no-mount` feature,
  Windows platform smoke installs WinFsp before compiling the workspace, stale
  Linux-only validation assertions now match the cross-platform mount boundary,
  and live-smoke scripts prebuild daemon/CLI binaries before waiting for mount
  exposure.
- The second remote CI pass is green for Rust, TypeScript, macOS/Windows
  platform smoke, and all consolidated validation groups. The Windows live
  smoke still showed the mount CLI process exiting before the WinFsp drive was
  visible; the current checkpoint hardens mount-session shutdown handling so a
  failed Ctrl-C handler install or disconnected shutdown channel does not
  terminate the mount process in non-interactive runner contexts, and expands
  the Windows live-smoke script to print daemon, mount, process, and WinFsp
  service diagnostics on failure.
- Windows live-smoke diagnostics then isolated the next failure to mount-process
  termination before any CLI stdout/stderr, with the daemon and WinFsp Launcher
  service both healthy. The Windows adapter now calls `winfsp_wrs::init()`
  before using WinFsp APIs so the delayed WinFsp DLL is loaded from the
  installed WinFsp directory instead of crashing at first API touch.
- The next Windows live-smoke pass confirmed that process termination still
  happened before CLI stdout/stderr. The root cause is the missing MSVC
  delayload linker wiring required by `winfsp_wrs`: without
  `/DELAYLOAD:winfsp-x64.dll`, the Windows loader attempts to resolve the
  WinFsp DLL before `main`, so `winfsp_wrs::init()` never gets a chance to load
  the installed runtime DLL. `operon-cli` and `operon-mount` now use
  `winfsp_wrs_build` build scripts on Windows to configure the delayed WinFsp
  DLL load path.
- The follow-up Windows live-smoke run reached a live mounted process and CLI
  output, but the mounted drive did not expose `seed.txt`. The Windows adapter
  now has opt-in `OPERON_MOUNT_TRACE` callback diagnostics and the Windows live
  smoke prints drive/root directory state so the next run can distinguish
  WinFsp mount visibility from directory callback behavior.
- The next Windows live-smoke diagnostics confirmed that `O:\` appears in
  `fsutil fsinfo drives`, but `dir O:\` returns `Incorrect function`. The
  smoke diagnostics now tolerate native diagnostic command failures and print
  daemon/mount logs before probing the mounted drive so callback trace output is
  preserved.
- The current Windows live-smoke diagnostic checkpoint adds tracing for the
  WinFsp volume-info and dispatcher-stopped callbacks, and records `mountvol`
  plus `fsutil fsinfo drivetype/volumeinfo` output. This should distinguish
  mount-manager exposure from callback dispatch and volume metadata failures
  before changing adapter behavior.
- Windows live-smoke diagnostics showed the drive letter is registered as a
  fixed drive, but volume information fails before `get_volume_info` reaches the
  adapter. The Windows WinFsp volume parameters now enable double buffering to
  match the more conservative default used by the `winfsp` Rust host wrapper.
- The manual live-smoke workflow now has per-job timeouts, and the macOS smoke
  script has a watchdog that dumps daemon, mount, process, and mounted-volume
  state before terminating a hung run.
- After double buffering did not change Windows live-smoke behavior, the
  Windows smoke build now enables a diagnostic-only `operon-mount/winfsp-debug`
  feature that forwards `winfsp_wrs/debug` so the next failed run can include
  WinFsp dispatcher/debug output without changing normal release builds.
- WinFsp debug output showed root directory opens arrive at the adapter as
  `Create "" FILE_OPEN` and were returning `STATUS_INVALID_DEVICE_REQUEST`
  before directory enumeration. The Windows adapter `create` path now stats and
  opens existing paths before falling back to mkdir/create-file behavior for
  missing paths.
- The next Windows live-smoke run still returned `STATUS_INVALID_DEVICE_REQUEST`
  for root opens while `OPERON_MOUNT_TRACE` showed no adapter `create` or
  `open` callback entry. The Windows adapter now exposes `CreateEx` as well as
  `Create`, delegating both paths to the same existing-path open-or-create
  logic required by newer WinFsp dispatch.
- Follow-up CI validation exposed a long-standing script fragility: completion
  generation was piped directly into `grep -q`, so `grep` could close the pipe
  early and trigger a `clap_complete` broken-pipe panic. The v0.8.1 integration
  coverage and v0.8 agent-skills validation scripts now write completion output
  to temporary files before asserting expected content.
- The latest manual live-smoke run confirmed baseline CI is green, while
  Windows still returns `STATUS_INVALID_DEVICE_REQUEST` for root opens before
  any adapter callback trace reaches stderr, and macOS prints `mount did not
  expose seed file` without enough immediate diagnostics. The next checkpoint
  makes macOS failure cleanup bounded and diagnostic-first, reduces Windows
  seed-exposure wait time, logs Windows callback flags and callback entry before
  remote filesystem calls, and narrows Windows creation dispatch to the
  `CreateEx` callback surface used by newer WinFsp examples.
- Follow-up live-smoke evidence confirmed the Windows failure happens before
