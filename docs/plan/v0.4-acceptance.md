# v0.4 Acceptance

v0.4 is scoped to:

- Runtime API stabilization
- Service / port capability
- Trace and audit CLI UX

It deliberately does not include Web Console, clipboard, screen/input, port
forwarding, proxying, VPN behavior, or new provider discovery beyond LAN mDNS.

## Validation

Run the canonical v0.4 validation:

```bash
pnpm install --frozen-lockfile
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
scripts/verify-v0.4-docker.sh
```

The Docker validation starts two reachable `operond` nodes and verifies:

- token-authenticated daemon calls
- capability discovery
- filesystem read/write/list/stat and streaming file transfer
- job execution, cancellation, timeout, stdout/stderr logs, streaming logs, and stdin
- policy denial for path escape, timeout, secrets, and unknown services
- scoped secret injection without exposing secret values
- service list/check for configured TCP services
- audit filters by capability, action, allowed state, and resource
- trace human summary mode and JSON mode
- LAN mDNS discovery
- read-only mount PoC
- example execution graph

## API Acceptance

Daemon errors should return:

```json
{
  "code": "forbidden",
  "message": "service `missing` denied by policy",
  "status": 403,
  "capability": "service:default",
  "resource": "missing"
}
```

The CLI and SDK should both parse the same structured error shape. Existing
callers that only expect `code` and `message` should keep working.

## Service Capability Acceptance

Configured services can be listed:

```bash
operon --config examples/docker-nodes.yaml service list node-a
```

Configured services can be checked:

```bash
operon --config examples/docker-nodes.yaml service check node-a daemon
```

Unknown services fail through policy:

```bash
operon --config examples/docker-nodes.yaml service check node-a missing
```

The service capability is metadata and health checking only. It does not proxy
traffic, forward ports, or create network reachability.

## Trace and Audit Acceptance

Audit can be filtered without shell post-processing:

```bash
operon --config examples/docker-nodes.yaml audit show node-a --capability service:daemon --action check --allowed true --resource daemon --limit 5
```

Trace commands have human and JSON modes:

```bash
operon trace show /tmp/operon-docker-trace.json
operon --json trace show /tmp/operon-docker-trace.json
```

`trace list` should only include trace-like JSON files, not every JSON file in a
directory.

## CI

`.github/workflows/ci.yml` must run on pull requests and pushes to every branch.
The Docker job must execute `scripts/verify-v0.4-docker.sh`.
