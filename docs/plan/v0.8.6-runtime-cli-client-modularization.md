# v0.8.6 Runtime, CLI, and Client Modularization

Status: Completed.

## Context

The first v0.8.4 modularization pass extracted daemon filesystem handlers,
pagination, CLI output helpers, target parsing, and CLI filesystem commands.
That reduced the immediate blast radius of the `ReadFileRange` work, but the
main runtime, CLI entrypoint, and Linux mount adapter still carry too many
unrelated responsibilities.

This phase is the follow-up maintainability pass before endpoint discovery UX. It
should preserve current behavior while making the next service, discovery, and
agent-facing changes cheaper and safer.

## Goals

- Keep [`operond/src/main.rs`](../../crates/operond/src/main.rs) focused on daemon startup and gRPC method routing.
- Keep [`operon-cli/src/main.rs`](../../crates/operon-cli/src/main.rs) focused on Clap model construction and
  high-level dispatch.
- Move reusable Rust gRPC client concerns out of CLI and mount code.
- Split the Linux mount adapter into module boundaries that match its runtime
  responsibilities.
- Align the TypeScript SDK public API with the core CLI/protocol surface.
- Reduce repeated validation-script setup without changing validation coverage.

## Work Items

### 1. Daemon Runtime Split

Extract behavior-preserving daemon modules:

- [`state.rs`](../../crates/operond/src/state.rs): `AppState`, initialization helpers, shared runtime handles.
- `runtime_service.rs`: `GrpcRuntime`, tonic stream type aliases, and gRPC
  method routing.
- [`auth.rs`](../../crates/operond/src/auth.rs): bearer auth and run/step metadata extraction.
- [`job_runtime.rs`](../../crates/operond/src/job_runtime.rs): job start, spawn, process group termination, cancellation,
  stdin pump, and completion.
- `job_logs.rs`: log buffer, snapshots, stream events, retention, and store
  append helpers.
- [`service_forward.rs`](../../crates/operond/src/service_forward.rs): TCP service authorization, health checks, and byte
  tunnel stream.
- `service_datagram.rs`: UDP datagram tunnel session lifecycle.
- [`audit.rs`](../../crates/operond/src/audit.rs): audit event construction, request context propagation, and store
  append.

The split must not change gRPC schemas, audit event fields, job status
semantics, process termination behavior, log retention limits, or service
forwarding behavior.

### 2. CLI Command Family Split

Extract non-filesystem CLI command families:

- [`commands/node.rs`](../../crates/operon-cli/src/commands/node.rs)
- [`commands/capability.rs`](../../crates/operon-cli/src/commands/capability.rs)
- [`commands/audit.rs`](../../crates/operon-cli/src/commands/audit.rs)
- [`commands/service.rs`](../../crates/operon-cli/src/commands/service.rs)
- [`commands/job.rs`](../../crates/operon-cli/src/commands/job.rs)
- [`commands/config.rs`](../../crates/operon-cli/src/commands/config.rs)
- [`commands/init.rs`](../../crates/operon-cli/src/commands/init.rs)
- [`commands/trace.rs`](../../crates/operon-cli/src/commands/trace.rs)
- [`commands/mount.rs`](../../crates/operon-cli/src/commands/mount.rs)

The entrypoint may keep the Clap command model and top-level dispatch in this
phase, but command handlers and domain-specific renderers should move out of
`main.rs`.

Add a shared rendering pattern where practical so text, JSON, and quiet output
contracts do not drift. This phase should specifically preserve:

- global `--json` produces one valid JSON document for scriptable commands.
- global `--quiet` suppresses human output.
- failed non-detached `job run` exits non-zero.
- raw byte streams still stay raw in non-JSON paths.

### 3. Shared Rust gRPC Client Crate

Add a lightweight Rust crate, tentatively [`operon-grpc-client`](../../crates/operon-grpc-client), for CLI and
mount reuse.

It should own:

- `grpc://` and `grpcs://` endpoint normalization into tonic-compatible URIs.
- bearer auth metadata injection.
- optional run/step request context metadata injection.
- typed `OperonRuntimeClient<Channel>` construction.
- Rust-side write/stdin chunk envelope helpers.

It should not absorb endpoint discovery UX or network mesh concerns; those stay
outside the shared client helper.

### 4. Linux Mount Adapter Split

Split [`operon-mount/src/lib.rs`](../../crates/operon-mount/src/lib.rs) into focused modules:

- [`remote_client.rs`](../../crates/operon-mount/src/remote_client.rs): `GrpcRemoteFs`, tonic client calls, request metadata.
- [`inode_table.rs`](../../crates/operon-mount/src/inode_table.rs): inode entry/table state and subtree updates.
- [`fuse_fs.rs`](../../crates/operon-mount/src/fuse_fs.rs): FUSE callback implementation.
- [`path.rs`](../../crates/operon-mount/src/path.rs): remote path normalization and child-name validation.
- [`errors.rs`](../../crates/operon-mount/src/errors.rs): errno/status mapping helpers.
- [`session.rs`](../../crates/operon-mount/src/session.rs): mount options, mount session, mount point setup.

Keep [`operon-mount`](../../crates/operon-mount) Linux-only and keep `RemoteFs` as the core mount adapter
boundary.

### 5. CLI Surface Adjustments

Add clearer aliases without removing existing commands:

- `operon graph run <workflow>`
- optionally `operon workflow run <workflow>`

Keep top-level `operon run <workflow>` as a compatibility alias for now.

Improve `fs read --output` JSON behavior so `operon --json fs read ... --output
file` returns a structured summary such as:

```json
{
  "path": "/remote.txt",
  "output": "/local.txt",
  "bytes_written": 123
}
```

### 6. TypeScript SDK Surface Alignment

Expose direct SDK methods for core protocol capabilities that are currently
only internal or missing:

- `statFs`
- `listFs`
- `runJob`
- `getJob`
- `cancelJob`
- `listCapabilities`
- `listAudit`

Keep existing graph runner methods and convenience methods compatible.

### 7. Validation Script Helpers

Extract common shell helpers for validation scripts where the duplication is
clear and low-risk:

- temporary config/workspace setup.
- daemon start and cleanup.
- `wait_for_node`.
- common trap cleanup behavior.

Do not reduce the current validation matrix or remove targeted phase scripts.

## Done When

- [`crates/operond/src/main.rs`](../../crates/operond/src/main.rs) no longer directly owns job runtime, job log
  retention, audit append, TCP service tunnel, or UDP datagram tunnel internals.
- [`crates/operon-cli/src/main.rs`](../../crates/operon-cli/src/main.rs) no longer owns non-fs command handlers and
  renderers.
- CLI and mount share Rust gRPC endpoint/auth/client helpers.
- [`operon-mount`](../../crates/operon-mount) has module boundaries for remote client, inode table, FUSE
  callbacks, paths, errors, and session lifecycle.
- TypeScript SDK exposes direct public methods for the listed core protocol
  capabilities.
- `fs read --output --json` returns a structured summary.
- graph/workflow aliases are documented and tested.
- existing Rust, TypeScript, service forwarding, mount, SDK, and integration
  validations remain green.

## Completed

- Added [`operon-grpc-client`](../../crates/operon-grpc-client) for shared Rust endpoint normalization, typed
  tonic client creation, auth/context metadata, and write/stdin stream chunk
  envelopes.
- Migrated CLI and Linux mount client paths to the shared Rust gRPC helper.
- Split non-filesystem CLI command handlers into `commands/*` modules and kept
  `main.rs` focused on Clap model construction and dispatch.
- Added `operon graph run` and `operon workflow run` aliases while preserving
  top-level `operon run`.
- Updated `fs read --output --json` to return a structured summary with path,
  output path, and bytes written.
- Split Linux mount code into remote client, inode table, FUSE callbacks, path,
  errors, and session modules.
- Split daemon auth, audit, state, job runtime/log retention, and service
  forwarding internals out of [`operond/src/main.rs`](../../crates/operond/src/main.rs).
- Exposed direct TypeScript SDK methods for capabilities, fs stat/list, job
  run/get/cancel, and audit listing.
- Added [`scripts/lib/validation.sh`](../../scripts/lib/validation.sh) for reusable low-risk validation helpers.
- Added [`scripts/verify-v0.8.6-runtime-cli-client-modularization.sh`](../../scripts/verify-v0.8.6-runtime-cli-client-modularization.sh) and wired
  it into CI.

## Remaining

- `GrpcRuntime` tonic method routing remains in [`operond/src/main.rs`](../../crates/operond/src/main.rs) for now;
  the internals it routes to are split into modules. A later cleanup can move
  the service impl into `runtime_service.rs` if the routing layer grows again.
- `job_logs` and UDP datagram forwarding are currently split by responsibility
  inside [`job_runtime.rs`](../../crates/operond/src/job_runtime.rs) and [`service_forward.rs`](../../crates/operond/src/service_forward.rs); separate files can be added
  later if those sections continue to grow.
