# v0.5 Acceptance

Status: Completed.

v0.5 is scoped to the gRPC runtime protocol migration.

It includes:

- protobuf contract finalization for the current runtime API.
- generated Rust protocol bindings.
- tonic gRPC daemon server.
- gRPC paths for node metadata, capability discovery, fs, jobs, audit, scoped
  secret use through jobs, and service checks.
- CLI execution graphs run over gRPC when their configured node endpoints are
  `grpc://` endpoints.
- streaming fs read/write and job event paths.
- CLI support for `grpc://` endpoints.
- TypeScript SDK support for gRPC endpoints through `nice-grpc`.
- HTTP/JSON facade kept for scripts, SDK compatibility, and debugging.

It deliberately does not include HTTPS/mTLS hardening, a new network layer,
port forwarding, proxying, graphical UI, or mount behavior changes.

## Validation

The canonical v0.5 validation should run:

```bash
pnpm install --frozen-lockfile
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
scripts/verify-v0.5-docker.sh
```

The Docker validation must prove:

- two `operond` nodes expose reachable gRPC endpoints.
- token-authenticated gRPC calls succeed.
- unauthorized gRPC calls fail with stable status/error metadata.
- capability discovery works over gRPC.
- fs stat/list/read/write work over gRPC.
- fs read and write use streaming semantics.
- job run/status/logs/stdin work over gRPC where applicable.
- policy denial behavior matches the HTTP facade.
- audit and trace records are produced for gRPC operations.
- service list/check works over gRPC.
- existing HTTP facade commands still work during the migration.

## Known Limits

- HTTP/JSON remains supported as a facade and compatibility layer.
- The TypeScript SDK uses `nice-grpc` for `grpc://`/`grpcs://` endpoints and
  keeps HTTP facade calls for `http://` endpoints.
- TLS and signed node identity can remain later hardening work unless v0.5
  explicitly expands scope.
