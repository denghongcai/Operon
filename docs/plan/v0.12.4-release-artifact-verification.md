# v0.12.4 Release Artifact Verification

Status: Completed.

## Goal

Close the loop between source-tree validation and public release usability by
verifying published GitHub Release artifacts after they are created.

## Context

v0.12 expanded draft release artifacts to Linux, macOS, and Windows core
runtime preview binaries. The release workflow builds, packages, smokes
binaries, and generates checksums before creating the draft release. The next
gap is post-publication validation: users consume release assets, not the local
build tree.

README Quickstart and release artifact validation should be checked against the
actual public assets before a release is declared complete.

## Scope

1. Release asset download verification.
   - Add a script or workflow entry that targets a GitHub Release tag.
   - Download `SHA256SUMS` and all expected Rust/SDK archives.
   - Verify every listed checksum and fail on missing or unexpected artifacts.

2. Cross-platform smoke commands.
   - Run `operon --version` and `operond --version` from downloaded assets.
   - Run `operon --help`, `operon doctor --help`, and `operon exec --help`.
   - Keep Linux FUSE and GLIBC baseline validation separate from macOS/Windows
     core runtime preview checks.

3. README Quickstart validation.
   - Extend `scripts/verify-readme-quickstart-docker.sh` or add a companion
     release verification script so README install commands stay aligned with
     actual asset names.
   - Document the exact command maintainers should run after publishing a
     release.

4. CI/release documentation.
   - Document whether verification is manual, scheduled, or invoked by
     `workflow_dispatch`.
   - Update README, `DEVELOPMENT.md`, AGENTS guidance, and release plan notes.

## Non-Goals

- No npm or crates.io publishing automation.
- No code-signing, notarization, or installer work.
- No OS package manager integration.
- No release promotion automation unless a separate phase defines the release
  state machine.

## Acceptance Criteria

- A maintainer can verify a public release tag from downloaded GitHub Release
  assets with one documented command or workflow dispatch.
- Checksum validation covers every expected archive.
- Linux, macOS, and Windows core runtime preview archives are smoke-tested from
  extracted release assets.
- README Quickstart artifact names match the release workflow.
- `docs/plan/development-phases.md` records the completed verification path and
  any manual release steps that remain.

## Completed

- Added `scripts/verify-release-artifacts.sh` to download public GitHub Release
  assets for a tag, verify `SHA256SUMS`, enforce the expected Linux/macOS/
  Windows/SDK artifact set, and smoke-test the current platform archive.
- Added the manual `Verify Release Artifacts` GitHub Actions workflow with
  Linux, macOS, and Windows runners.
- Added the manual `Verify README Quickstart` GitHub Actions workflow so
  README install-path validation runs against the public release tag in Actions
  instead of being accepted from a local workstation run.
- Added `scripts/verify-v0.12.4-release-artifact-verification.sh` and wired it
  into CI and DEVELOPMENT.md.
- Updated release documentation so maintainers use manual GitHub Actions
  workflows for public release artifact and README Quickstart verification
  after publishing.

## Remaining

- No v0.12.4 work remains.
- npm, crates.io, code-signing, notarization, installer automation, and release
  promotion automation remain outside this phase.
