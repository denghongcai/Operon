# v0.8.8 Filesystem Stream Handler Cleanup

Status: Completed.

Goal: keep full-file filesystem stream behavior inside the daemon filesystem
service module instead of the tonic runtime router.

Review finding:

- `crates/operond/src/main.rs` still owned `ReadFile` and `WriteFile` stream
  authorization, workspace path resolution, audit failure handling, chunk-size
  validation, and file IO.
- That left the daemon runtime router responsible for filesystem service logic
  even after v0.8.7 centralized the same helper pattern in `fs_service.rs`.

Completed:

- Added `fs_service::read_stream` and `fs_service::write_stream`.
- Reused the v0.8.7 authorization and workspace path helper boundary for
  full-file stream reads and writes.
- Reduced `operond/src/main.rs` `ReadFile` and `WriteFile` methods to runtime
  authorization, audit context scoping, and delegation.
- Added `scripts/verify-v0.8.8-fs-stream-handler-cleanup.sh`.

Validation:

```bash
scripts/verify-v0.8.8-fs-stream-handler-cleanup.sh
cargo fmt --check
cargo clippy --workspace --locked -- -D warnings
cargo test --workspace --locked
```

Remaining:

- No v0.8.8 work remains.
- Moving the full tonic `GrpcRuntime` trait implementation out of
  `operond/src/main.rs` remains a future maintainability candidate if runtime
  method routing grows again.
