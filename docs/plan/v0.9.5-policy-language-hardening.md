# v0.9.5 Policy Language Hardening

Status: Completed.

Goal: make Operon policy easier to reason about, explain, and audit before
adding new capability surfaces.

This phase follows v0.9.2 policy-derived capability discovery and v0.9.4
runtime hardening. It should improve the existing filesystem, job, service, and
secret policy model without turning Operon into a network policy engine.

## Scope

1. Shared policy decision model.
   - Introduce a small internal policy decision shape for allowed and denied
     checks.
   - Include stable fields for subject, capability id, action, resource, and a
     machine-readable reason code.
   - Keep human-readable deny messages, but derive them from the same decision
     path used by daemon authorization.

2. Policy explain output.
   - Extend `operon config explain` to show the effective policy surface in a
     way agents can consume.
   - Include effective filesystem grants, job cwd/timeout/env/secret limits,
     and service action grants.
   - Do not expose secret values.

3. Authorization consistency.
   - Align filesystem, job, service, and secret authorization paths around the
     shared decision vocabulary.
   - Preserve current allow/deny behavior unless a mismatch is found and
     explicitly documented.
   - Keep audit output stable enough for existing CLI filters while adding
     clearer denial reasons.

4. Config schema compatibility.
   - Keep the current `policy:` config shape working.
   - Allow additive policy metadata only when it has a clear runtime use.
   - Unknown config fields should continue to warn and not block startup.

5. Documentation and validation.
   - Document the policy vocabulary in README, `PROTOCOL.md`, and
     `docs/architecture/runtime-api.md` where user-facing behavior changes.
   - Add a dedicated validation script and CI matrix entry for v0.9.5.
   - Update agent-facing guidance in `AGENTS.md`.

## Non-Goals

- No VPN, mesh, NAT traversal, subnet, routing, or packet policy.
- No provider abstraction.
- No clipboard, screen, audio, remote desktop, or file sync capability.
- No new query database.
- No breaking replacement of the existing `policy:` config schema.
- No general-purpose expression language unless a concrete policy check needs
  it in this phase.

## Done When

- Policy decisions have a shared internal representation with stable reason
  codes.
- Filesystem, job, service, and secret deny paths use the shared decision
  vocabulary or have a documented reason for staying separate.
- `operon config explain --json` includes effective policy grants without
  leaking secrets.
- Human `operon config explain` output remains readable and names the effective
  grants and limits.
- Audit denial reasons are clearer and continue to support existing CLI
  filters.
- Existing policy config examples still parse.
- Unknown config field warnings remain non-blocking.
- [`scripts/verify-v0.9.5-policy-language-hardening.sh`](../../scripts/verify-v0.9.5-policy-language-hardening.sh) is added and wired into
  CI.
