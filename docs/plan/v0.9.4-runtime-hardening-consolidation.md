# v0.9.4 Runtime Hardening Consolidation

Status: Completed.

Goal: consolidate the remaining planned runtime hardening candidates into one
bounded phase before adding new product capability surfaces.

This phase intentionally groups several small deferred items so they can be
validated together as runtime semantics and maintainability hardening, rather
than producing many tiny follow-up phases.

## Scope

1. Service health semantics.
   - Make TCP and UDP service health results explicit and consistent.
   - Preserve the service boundary: health checks observe configured local
     services; they do not create network connectivity, routing, or relays.
   - Ensure audit events distinguish policy denial, unsupported protocol/action,
     local connection failure, timeout, and successful reachability.

2. Job log restart visibility.
   - Evaluate and implement restart-safe job log inspection using the existing
     append-only store records.
   - Keep job log APIs bounded and paginated/snapshot-oriented where
     appropriate.
   - Do not introduce a query database in this phase.

3. Workspace traversal hardening.
   - Add Linux `openat2` or fd-relative traversal where it materially reduces
     symlink/race risk for workspace filesystem operations.
   - Keep cross-platform behavior explicit and tested when Linux-only APIs are
     unavailable.

4. Shell-free job execution design and minimal protocol path.
   - Add a protocol-level `argv[]` job execution path if it can be introduced
     without breaking existing shell-command behavior.
   - Keep existing `command` shell execution compatible for CLI/script users.
   - Document when agents should prefer argv execution over shell execution.

5. Config UX cleanup.
   - Make the different LAN advertisement defaults in `operon init config` and
     `operon onboard` explicit in command output and docs.
   - Preserve the current defaults unless the implementation finds a concrete
     mismatch with the intended workflows.

6. Focused maintainability cleanup.
   - Move `GrpcRuntime` tonic routing into `runtime_service.rs` if it helps keep
     `operond/src/main.rs` focused.
   - Split UDP datagram state-machine code out of `service_forward.rs` only if
     the service health or job-log work causes that file to grow materially.
   - Split large local input streaming helpers in CLI only if argv/job-log work
     touches the same client paths.

## Non-Goals

- No VPN, mesh, NAT traversal, relay, global routing, or subnet-routing work.
- No browser management UI or CLI TUI console.
- No indexed query database or SQLite migration.
- No new screen, clipboard, audio, remote desktop, or sync capability.
- No breaking removal of existing shell-command job execution.
- No broad module rewrite unrelated to the scoped runtime hardening items.

## Done When

- TCP and UDP service health behavior is documented, audited, and covered by
  focused tests.
- Restart behavior for job log inspection is either implemented and validated
  or explicitly bounded with a documented reason and test coverage for the
  chosen behavior.
- Workspace traversal hardening has Linux-focused tests or a documented
  cross-platform fallback.
- The job execution API has a clear shell-command vs argv contract, with CLI,
  SDK, protocol docs, and tests aligned if argv execution is implemented.
- `init config` and `onboard` explain their LAN advertisement defaults.
- Any touched large daemon/CLI files are kept behind focused module boundaries.
- A new `scripts/verify-v0.9.4-runtime-hardening-consolidation.sh` validation
  gate is added and wired into CI.

## Suggested Execution Order

1. Write failing tests for service health/audit semantics.
2. Implement minimal service health semantic cleanup.
3. Write failing tests for job log restart visibility.
4. Implement store-backed job log loading or document and validate the bounded
   behavior if full reload is not appropriate.
5. Add Linux workspace traversal hardening tests.
6. Implement Linux hardening with explicit non-Linux fallback behavior.
7. Add protocol/CLI/SDK tests for argv execution.
8. Implement argv execution only if the compatibility contract remains small.
9. Update config UX text for `init config` and `onboard`.
10. Apply focused module splits only where the preceding tasks increased file
    responsibility.
11. Add the v0.9.4 validation script, README guidance, AGENTS update, and CI
    matrix entry.

- Added `operon_store::load_job_logs` and daemon startup seeding for bounded

- Added explicit workspace traversal hardening strategy coverage and symlink
  parent escape validation.
- Added shell-free job `argv[]` support in proto, Rust core/protocol, daemon,
  CLI, TypeScript SDK, generated TypeScript bindings, PROTOCOL.md, README, and
  runtime API docs.
- Bumped `PROTOCOL_VERSION` to `v0.9.4`.

  `operon onboard`.
- Added `scripts/verify-v0.9.4-runtime-hardening-consolidation.sh` and wired it
  into CI.
