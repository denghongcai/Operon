# v0.11.3 Platform Capability Matrix and CI Smoke

Status: Completed.

Goal: decide how far Operon can align macOS and Windows with the current Linux
runtime before expanding public release artifacts beyond Linux.

## Research Conclusion

macOS and Windows alignment is feasible for the core daemon and CLI runtime, but
not as full feature parity in the first distribution phase. The right first
target is a cross-platform core runtime preview:

- daemon startup, config loading, authentication, gRPC protocol, audit, trace,
  graph, capability discovery, diagnostics, filesystem RPCs, non-interactive
  exec, interactive exec sessions, and service health/forwarding should be made
  buildable and smoke-testable on macOS and Windows.
- Linux remains the only supported mount platform until explicit macFUSE and
  WinFsp adapter phases exist.
- Linux keeps the strongest workspace containment story because
  `openat2(RESOLVE_BENEATH)` is Linux-specific; macOS and Windows should expose
  clearly documented fallback containment semantics until platform-native
  handle-relative hardening is designed.
- Windows process-tree cancellation needs a Job Object based implementation for
  parity with Unix process-group termination. Until then, Windows cancellation
  should be documented as best-effort unless validation proves stronger
  behavior.
- Interactive exec should continue through the existing `portable-pty`
  abstraction. Validate `portable-pty` behavior on Windows CI before claiming
  Windows session support.

## Proposed Scope

1. CI platform smoke.
   - Add macOS and Windows GitHub Actions jobs for `cargo check --workspace
     --locked` and focused non-Linux-safe tests.
   - Keep Linux-only mount validation on Linux.
   - Add a static validation script that records the supported/candidate/deferred
     platform matrix.

2. Runtime compile and behavior fixes.
   - Gate Unix signal handling, file permission checks, and process-group logic
     behind explicit platform modules.
   - Keep Linux-only mount APIs behind unsupported-platform errors on non-Linux.
   - Make command-string exec defaults explicit per platform; prefer argv in
     docs and automation.

3. Interactive session validation.
   - Keep `portable-pty` as the only PTY abstraction.
   - Add macOS and Windows smoke coverage for workspace compilation, platform
     shell defaults, CLI terminal sizing, and core protocol/config/fs/exec
     packages.
   - Add real short-lived PTY session open/output/exit validation before
     publishing macOS or Windows release artifacts.

4. Documentation and release boundary.
   - Update README, `PROTOCOL.md`, architecture docs, and release docs to state
     which platforms are supported, candidate, or deferred.
   - If macOS/Windows binaries are not shipped yet, call the pre-1.0 public
     release Linux-only rather than implying cross-platform parity.

## Non-Goals

- No macFUSE or WinFsp implementation in this phase.
- No Windows Job Object cancellation implementation unless needed to pass a
  minimal cancellation contract.
- No new PTY backend dependency beyond `portable-pty`.
- No network/VPN/mesh responsibility expansion.
- No claim that macOS or Windows workspace containment is equivalent to Linux
  `openat2(RESOLVE_BENEATH)`.

## Done When

- The platform capability matrix is documented in the plan and architecture
  docs.
- CI has explicit macOS and Windows smoke coverage for the core workspace, or
  the release docs explicitly state Linux-only pre-1.0 support.
- Interactive exec documentation says `portable-pty` is the intended
  cross-platform abstraction.
- Mount support remains clearly Linux-only, with macFUSE and WinFsp deferred.
- README Quickstart, release workflow expectations, and validation scripts agree
  on the supported target set.

## Completed

- Documented the platform capability matrix in this plan and
  `docs/architecture/technology-and-protocol-decisions.md`.
- Added macOS and Windows Rust platform smoke CI for `cargo check --workspace
  --locked` and focused core package tests.
- Documented that public release artifacts remain Linux-only while macOS and
  Windows are candidate core runtime platforms.
- Kept Linux FUSE mount support Linux-only, with macFUSE and WinFsp deferred.
- Kept interactive exec validation on the existing `portable-pty` abstraction.
- Added platform-specific shell defaults for command-string exec/session
  requests: Unix uses `/bin/sh -c`, Windows uses `cmd.exe /C`.
- Added `scripts/verify-v0.11.3-platform-capability-matrix.sh` and wired it
  into CI and DEVELOPMENT.md.
- Follow-up CI repair authenticated `arduino/setup-protoc` with the workflow
  token to avoid unauthenticated GitHub API rate limits, made the preserved
  environment test accept Windows `Path` casing, and updated the older v0.8.6
  modularization verifier to match the current service TCP/datagram module
  boundaries.

## Remaining

- macOS and Windows release artifacts were intentionally deferred out of
  v0.11.3 and are addressed by the v0.12 release/distribution phase.
