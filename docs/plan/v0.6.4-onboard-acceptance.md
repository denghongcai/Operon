# v0.6.4 Onboard Acceptance

## Goal

v0.6.4 adds a fast interactive setup entrypoint without replacing the stable
command-style CLI configuration surface.

```text
v0.6.4 = guided first-run configuration over existing CLI primitives.
```

## Scope

- Add `operon onboard`.
- Generate ordinary unified config, token, and daemon start command files.
- Support daemon, client, and combined local setup roles.
- Support LAN discovery as an optional onboarding input.
- Support coarse capability preauthorization through policy generation.
- Print the equivalent command-style CLI steps so onboarding remains transparent.
- Keep command-style entrypoints such as `init config` and
  `node discover --provider lan` as the scriptable configuration interface.

## Non-goals

- No onboard-only config format.
- No remote daemon policy mutation.
- No Cloudflare, Tailscale, WireGuard, SSH, or Kubernetes API discovery.
- No VPN, relay, NAT traversal, or mesh configuration.
- No complex RBAC or admin API.

## Validation

```bash
cargo fmt --check
cargo test --workspace --locked
cargo check --workspace --locked
cargo clippy --workspace --locked -- -D warnings
scripts/verify-v0.6.4-onboard.sh
```

## Done When

- `operon onboard --yes --role both` creates a usable setup directory.
- Generated `config.yaml` and `token` are normal Operon config artifacts.
- A daemon can start with the generated policy and token.
- The CLI can use the generated node config to ping, inspect capabilities, run
  fs operations, and inspect audit.
- Existing command-style configuration commands remain available and documented.
