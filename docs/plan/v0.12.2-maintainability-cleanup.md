# v0.12.2 Maintainability Cleanup

Status: Planned.

Goal: perform behavior-preserving cleanup around the remaining large daemon and
CLI surfaces after release/platform hardening, so future feature work stays
localized and easier to validate.

## Scope

This phase should reduce high-change files by moving cohesive responsibilities
behind focused module boundaries. It should not change runtime protocol shape or
user-facing behavior unless a bug is found during extraction.

Current hotspots to reassess before implementation:

- `crates/operond/src/main.rs`
- `crates/operon-cli/src/grpc.rs`
- `crates/operond/src/exec_session.rs`
- `crates/operon-cli/src/commands/exec.rs`
- `crates/operon-cli/src/commands/config.rs`

Planned:

1. Daemon runtime router cleanup.
   - Move remaining RPC routing clusters out of `crates/operond/src/main.rs`
     where a cohesive service module already owns the behavior.
   - Keep `main.rs` focused on daemon startup, config loading, server wiring,
     auth context setup, and service delegation.

2. CLI gRPC base helper cleanup.
   - Continue extracting shared client request/stream setup from
     `crates/operon-cli/src/grpc.rs` into focused helper modules when a command
     family already has a clear boundary.
   - Preserve existing JSON, quiet, error, and script-facing CLI contracts.

3. Exec command/session UI cleanup.
   - Split command parsing, session terminal I/O, local TTY sizing, resize
     forwarding, and response rendering where those responsibilities are still
     coupled.
   - Keep protocol envelopes and SDK behavior unchanged.

4. Config command cleanup.
   - Separate config explain rendering, init/config writing helpers, and
     unknown-field warning presentation if the current file still mixes those
     responsibilities.
   - Preserve existing config file shape and `operon config explain` JSON
     output.

5. Validation.
   - Add a maintainability validation script that checks the new module
     boundaries and key public help paths.
   - Run full Rust and TypeScript validation because this phase touches shared
     command/runtime code.
   - Update `docs/plan/development-phases.md` with exact files moved and
     remaining maintainability candidates.

## Non-Goals

- No protocol changes.
- No feature work.
- No CLI output changes except where required to fix a bug discovered during
  extraction.
- No broad rename churn.
- No cleanup of unrelated crates just to reduce line counts.

## Done When

- The daemon router and CLI client responsibilities are smaller and clearer.
- Existing behavior is preserved by focused tests and repository validation
  scripts.
- The phase log records what was extracted, what validation passed, and which
  hotspots remain.

## Remaining

- Entire phase remains to be implemented.
