# v0.13.7 Mount Adapter Strategy

Status: Completed.

## Goal

Decide whether and how Operon should pursue macFUSE and WinFsp mount adapters
without blurring the existing boundary between core filesystem RPCs and
platform-specific live mount integrations.

## Context

Linux FUSE support exists as the only supported mount adapter. macOS and
Windows release artifacts now cover core daemon and CLI workflows, but their
filesystem support is the Core FS Protocol through gRPC, CLI, and SDK access,
not a live OS mount layer.

Implementing macFUSE or WinFsp would add platform dependencies, permission
requirements, packaging complexity, license review, and CI challenges. The
useful v0.13.7 output is therefore a strategy decision and follow-up sequence,
not immediate adapter implementation.

## Product Boundary Decision

Mount adapters remain optional convenience layers over the Core FS Protocol.
They do not own policy, audit, durable metadata, conflict resolution, transport,
or network reachability.

The Core FS Protocol remains the supported cross-platform filesystem surface for
macOS and Windows until an adapter reaches explicit support status. README,
architecture docs, CLI help, and release notes must not imply macFUSE or WinFsp
support before implementation and validation exist.

File sync, offline caching, global workspace replication, File Provider-style
document synchronization, and remote desktop remain out of scope. If Operon
later pursues a sync product, that must be a separate product decision instead
of being smuggled into the mount adapter layer.

## Recommended Implementation Sequence

1. Keep Linux FUSE as the only supported live mount before v1.0.
   - The current Linux adapter is real and validated.
   - Adding non-Linux live mounts now would expand support burden more than it
     improves the core runtime.

2. Extract a shared mount-core boundary before any non-Linux adapter.
   - Move platform-neutral concerns behind a reusable boundary:
     `RemoteFs`, path validation, stat/list/read/write/range-read operation
     mapping, error vocabulary, precondition/version handling, audit-facing
     request metadata, and cancellation semantics.
   - Keep Linux-specific FUSE callback types in the Linux adapter.
   - This reduces the risk of duplicating policy/error behavior across macFUSE
     and WinFsp.

3. Attempt macFUSE FSKit as the first experimental non-Linux adapter.
   - Rationale: it can likely reuse more of the existing `fuser` callback shape
     and avoids the kernel-extension security downgrade path.
   - Initial support must be explicitly experimental and limited to macOS 15.4+
     with `backend=fskit`.
   - Kernel backend support should not be the default because of kext approval,
     reduced-security requirements on Apple silicon, restart flow, and CI
     friction.

4. Treat WinFsp as a later native Windows adapter phase.
   - Rationale: Windows filesystem semantics are not POSIX semantics. The
     WinFsp native API is a better fit for ACLs, delete semantics, reparse
     points, UTF-16 names, async I/O, and Windows service lifecycle.
   - Before implementation, decide whether GPLv3 `winfsp-rs` and WinFsp's
     licensing model are acceptable for this Apache-2.0 workspace or whether
     Operon needs direct FFI, an optional separately licensed component, or a
     commercial license path.

## Expected Capability Cuts

Non-Linux adapters should start with a conservative subset:

- required: stat, list, read, read range, write range, truncate, mkdir, delete,
  rename, permission-denied mapping, cancellation on unmount, and read-after-write.
- required: daemon-owned policy and audit decisions for every remote operation.
- required: clear diagnostics when the platform runtime is missing or the
  adapter cannot mount.
- deferred: offline cache, conflict resolution, file watching or invalidation
  notifications, hard links, extended attributes, Windows alternate data
  streams, full Windows security descriptor editing, POSIX uid/gid/mode
  fidelity on Windows, Finder/Explorer shell polish, and global background sync.

macFUSE FSKit-specific cuts:

- mount points outside `/Volumes` are not part of the first FSKit path.
- FUSE notification API support is not assumed.
- `fuse_context_t`-dependent behavior is not assumed.
- mount options handled by the kernel backend are not assumed.
- performance must be treated as lower confidence than the kernel backend until
  measured.

WinFsp-specific cuts:

- do not promise POSIX mode/uid/gid fidelity.
- do not expose Windows ACL editing or alternate data streams in the first
  adapter.
- do not require machine-wide drive mapping in the first version; a foreground
  or user-scoped service-style smoke path is acceptable.

## Validation Plan

Shared mount-core validation:

- unit tests for path validation, operation mapping, errno/status mapping,
  version/precondition propagation, and cancellation.
- daemon integration tests for policy denial, audit recording, read/write/range
  behavior, and list consistency.

Linux validation:

- keep current `linux-system` validation scripts as the live mount gate.
- keep focused `operon-mount` unit tests for logic that does not require a live
  kernel mount.

macFUSE FSKit experimental validation:

- compile on macOS in normal CI.
- run non-mount unit tests on GitHub-hosted macOS.
- add a manual or self-hosted macOS smoke gate for live mount behavior until
  hosted runner behavior is proven stable.
- smoke cases: mount, list, stat, read, random read, write, truncate, rename,
  delete, permission denied, unmount/cancellation.

WinFsp experimental validation:

- compile on Windows in normal CI.
- install WinFsp in a Windows validation job only after packaging/licensing is
  accepted.
- prefer native WinFsp API tests over FUSE-compatibility tests.
- smoke cases: mount or drive exposure, list, stat, read, random read, write,
  truncate, rename, delete, permission denied, unmount/service stop, and missing
  WinFsp runtime diagnostics.

## Decision

Operon should remain Linux-only for supported live mounts before v1.0. The next
mount-related engineering phase should not implement macFUSE or WinFsp directly.
It should first extract a shared mount-core boundary and validation harness.

After that boundary exists, the first non-Linux implementation candidate should
be an experimental macFUSE FSKit adapter. WinFsp should be planned after the
macFUSE experiment, and should prefer WinFsp's native API unless a short spike
proves the FUSE compatibility layer is sufficient without hiding important
Windows semantics.

Decision update: this Linux-only pre-v1.0 support boundary is superseded by
[`v0.14 Cross-Platform Live Mount`](v0.14-cross-platform-live-mount.md). The
dependency and validation research in this document remains useful, but live
mount is now treated as a core capability that must reach macOS and Windows.
