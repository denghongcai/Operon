# v0.13 Mount Adapter Strategy

Status: Planned.

## Goal

Decide whether and how Operon should pursue macFUSE and WinFsp mount adapters
without blurring the existing boundary between core filesystem RPCs and
platform-specific live mount integrations.

## Context

Linux FUSE support exists as the only supported mount adapter. macOS and
Windows core runtime previews now exist, but their filesystem support is the
core gRPC filesystem API, not a live mount layer.

Implementing macFUSE or WinFsp would add platform dependencies, permission
requirements, packaging complexity, and CI challenges. The next useful step is
a strategy phase, not immediate adapter implementation.

## Scope

1. Capability and dependency matrix.
   - Document macFUSE and WinFsp install prerequisites, permission model,
     kernel/driver requirements, signing or admin requirements, and CI
     feasibility.
   - Compare those against the current Linux FUSE adapter behavior.

2. Product boundary decision.
   - Decide whether mount adapters remain optional convenience layers over the
     Core FS Protocol.
   - Keep file sync, offline caching, and global workspace replication out of
     scope unless separate decision docs intentionally add them.

3. Implementation options.
   - Define possible adapter paths:
     - continue Linux-only mount support for pre-1.0.
     - add macFUSE first and keep WinFsp deferred.
     - add WinFsp first and keep macFUSE deferred.
     - build a shared mount adapter abstraction before adding new platforms.
   - Record tradeoffs for packaging, validation, support burden, and user
     expectations.

4. Validation plan.
   - Define what must be testable in CI and what must remain manual.
   - Define acceptance criteria for read, write, range read, directory listing,
     error mapping, cancellation, and permission-denied behavior.

## Non-Goals

- No macFUSE implementation in this strategy phase.
- No WinFsp implementation in this strategy phase.
- No file sync engine, offline cache, or remote desktop functionality.
- No change to the Core FS Protocol unless the strategy identifies a concrete
  protocol gap that must be planned separately.

## Acceptance Criteria

- A documented platform strategy states whether mount support remains Linux-only
  for pre-1.0 or which non-Linux adapter should be attempted first.
- The strategy includes dependency, packaging, CI, and support implications.
- README, architecture docs, and AGENTS guidance continue to describe non-Linux
  filesystem support as core RPC access unless a later implementation phase
  changes that.
- `docs/plan/development-phases.md` records the decision and the next concrete
  mount-related phase, if any.
