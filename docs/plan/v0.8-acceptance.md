# v0.8 Acceptance

Status: Completed.

v0.8 is scoped to an Agent Skills Pack.

It includes:

- CLI help review for all public commands that agents will use.
- a CLI config interpretation view, such as `operon config explain`.
- repo-local skills under `skills/<name>/SKILL.md`.
- skills that teach agents how to use `operon` CLI, `config.yaml`,
  `PROTOCOL.md`, and the TypeScript SDK.
- scenario guidance for node inspection, capability inspection, fs, jobs,
  services, audit, traces, and execution graphs.
- instructions that agents should use `operon <command> --help` for exact flag
  syntax instead of treating skills as a duplicated CLI manual.
- explicit safety guidance for destructive or policy-sensitive operations.
- examples for TCP service forwarding and UDP/datagram forwarding.
- guidance to verify actions through audit and trace output.

Implemented artifacts:

- `operon config explain` with text and JSON output.
- repo-local skills in `skills/*/SKILL.md`.
- CLI help coverage for public command paths used by agents.
- `scripts/verify-v0.8-agent-skills.sh`.
- CI matrix coverage for v0.8 validation.

It deliberately does not include MCP, a separate agent runtime, a new server, a
parallel control plane, browser UI, or bypasses around Operon policy.

## Validation

The canonical v0.8 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
scripts/verify-v0.8-agent-skills.sh
```

The skills validation must prove:

- every skill has a `SKILL.md`.
- all public CLI command paths expose working `--help`.
- the config interpretation view summarizes active daemon, client, auth,
  policy, service, and secrets configuration without requiring raw YAML
  inspection.
- skill descriptions and trigger guidance are present.
- skills point to existing CLI, SDK, and protocol surfaces.
- skills explain scenarios and command choice, while detailed syntax remains in
  CLI help.
- skills do not reference MCP as an Operon runtime surface.
- destructive workflows require explicit confirmation language.
- documented examples use current command names, including `service forward`
  and `service forward-udp`.
- audit and trace verification is included for mutating workflows.

## Known Limits

- v0.8 teaches agents how to use Operon; it does not add new runtime APIs.
- Skills can include direct protocol or SDK guidance, but the CLI remains the
  preferred human and agent-operable surface for common workflows.
- Destructive operations should remain opt-in and policy-gated.
