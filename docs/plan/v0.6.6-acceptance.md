# v0.6.6 Acceptance

v0.6.6 is the release-hardening checkpoint before v0.7 TUI work.

## Scope

- Workspace filesystem operations must reject symlink-resolved paths that escape
  the configured daemon workspace.
- Job execution must clear the inherited daemon environment and inject only
  `job.env_allowlist` variables plus authorized requested secrets.
- Execution graph calls must propagate `x-operon-run-id` and
  `x-operon-step-id` metadata into daemon audit events.
- CLI file reads and followed job logs must consume gRPC streams chunk by chunk.
- The TypeScript SDK must expose a true file stream API while retaining
  `readFileBytes`.
- Daemon code should use runtime crate APIs for filesystem policy/path
  handling, job policy/environment construction, append-only store operations,
  and service health checks.

## Validation

Run before tagging `v0.6.6`:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm --filter @operon/sdk typecheck
pnpm --filter @operon/sdk test
pnpm --filter @operon/sdk build
scripts/verify-v0.5-docker.sh
scripts/verify-v0.6.3-fs-copy.sh
scripts/verify-v0.6.4-onboard.sh
```

## Release

After validation passes:

```bash
git tag v0.6.6
git push origin v0.6.6
```

The pushed tag triggers the draft GitHub release workflow.

## Known Limits

- Filesystem writes still do not provide conflict detection, etags, locks, or
  compare-and-swap preconditions.
- `grpcs://` endpoint naming is reserved by the client, but full TLS identity
  policy remains a later decision.
- Runtime crate boundaries are intentionally helper-level in this release; the
  daemon is not fully decomposed into small runtime modules yet.
