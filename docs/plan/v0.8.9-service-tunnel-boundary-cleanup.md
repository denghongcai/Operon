# v0.8.9 Service Tunnel Boundary Cleanup

Status: Completed.

Goal: keep service tunnel target parsing, authorization, protocol checks, audit
handling, and connection setup inside the daemon service forwarding module.

Review finding:

- [`crates/operond/src/main.rs`](../../crates/operond/src/main.rs) still owned TCP and UDP tunnel open handshakes:
  target-envelope validation, service policy authorization, protocol mismatch
  checks, audit records, TCP connection setup, and datagram stream delegation.
- That kept service forwarding business logic in the tonic runtime router
  instead of behind [`service_forward.rs`](../../crates/operond/src/service_forward.rs).

Completed:

- Added `service_forward::open_service_tunnel` and
  `service_forward::open_service_datagram_tunnel`.
- Added service tunnel stream type aliases for the runtime trait associated
  stream types.
- Reduced [`operond/src/main.rs`](../../crates/operond/src/main.rs) service tunnel RPC methods to gRPC auth, audit
  context scoping, and delegation.
- Added [`scripts/verify-v0.8.9-service-tunnel-boundary-cleanup.sh`](../../scripts/verify-v0.8.9-service-tunnel-boundary-cleanup.sh).

Validation:

```bash
scripts/verify-v0.8.9-service-tunnel-boundary-cleanup.sh
cargo fmt --check
cargo clippy --workspace --locked -- -D warnings
cargo test --workspace --locked
```

Remaining:

- No v0.8.9 work remains.
- Moving the full tonic `GrpcRuntime` trait implementation out of
  [`operond/src/main.rs`](../../crates/operond/src/main.rs) remains a future maintainability candidate if runtime
  method routing grows again.
