# v0.8.3 Read Range and Release Cleanup

Status: Completed.

## Goal

Close the concrete mount performance gap and make release, package, and protocol
version semantics explicit before endpoint discovery UX work resumes.

This phase is intentionally narrow: it changes the filesystem protocol surface
for efficient range reads and cleans up version/documentation drift. It should
not refactor daemon or CLI module structure.

## Scope

- Add a gRPC `ReadFileRange` RPC for direct byte-range file reads.
- Add protocol/core conversion types for range-read requests and responses.
- Implement daemon range reads with direct seek-and-read behavior.
- Update the Linux FUSE mount adapter to use `ReadFileRange` instead of
  streaming the full file and skipping bytes locally.
- Add CLI/SDK helpers only where they support the protocol contract or tests;
  do not make range reads a primary human CLI workflow unless there is a clear
  operational use case.
- Document the relationship between:
  - GitHub release tags.
  - Rust crate package versions.
  - TypeScript SDK package versions.
  - `PROTOCOL_VERSION`.
- Update README release examples so the install path does not imply a stale
  fixed release.

## Non-Goals

- No service or endpoint discovery UX work.
- No large `operond` or `operon-cli` module split.
- No FUSE page-cache or read-ahead redesign beyond consuming `ReadFileRange`.
- No npm/crates.io publishing automation unless separately planned.

## Validation

- Unit tests for range-read request validation and daemon read behavior.
- Mount-level test coverage proving `read_range(path, offset, size)` calls the
  range-read protocol path rather than full-file streaming.
- Protocol/SDK generation checks after proto changes.
- Documentation check that explains why `PROTOCOL_VERSION` can remain stable
  across skills, testing, or internal cleanup phases.
- Standard workspace validation:
  - `cargo fmt --check`
  - `cargo test --workspace --locked`
  - `cargo clippy --workspace --locked -- -D warnings`
  - `pnpm typecheck`
  - `pnpm test`

## Done When

- FUSE random reads no longer download file contents before the requested
  offset.
- `ReadFile` remains the streaming full-file API.
- `ReadFileRange` is documented as the efficient random-read API.
- README and release docs no longer hard-code a stale release version.
- Version policy is explicit enough that future phases know when to bump
  release, package, and protocol versions separately.

## Completed

- Added `ReadFileRange(FsReadRangeRequest) returns (FileChunk)` to
  `proto/operon/runtime.proto`.
- Added core/protocol range-read request conversions and TypeScript SDK
  generation.
- Implemented daemon direct seek-and-read range handling with offset/size
  validation and audit events.
- Updated the Linux FUSE `GrpcRemoteFs::read_range` path to call
  `ReadFileRange` instead of full-file `ReadFile`.
- Added SDK `readFileRangeBytes`.
- Bumped `PROTOCOL_VERSION` to `v0.8.3` because the gRPC schema changed.
- Updated README, `PROTOCOL.md`, and runtime architecture docs with range-read
  semantics and version policy.
- Added `scripts/verify-v0.8.3-read-range-release-cleanup.sh` and wired it into
  CI.
