# v0.14 macOS Live Mount Smoke Runbook

Status: blocked until a macOS host can complete FUSE-T network-volume mounts.

## Purpose

This runbook is the release-gate procedure for the remaining v0.14 macOS live
mount validation. macFUSE FSKit and kernel-backend attempts on GitHub-hosted
macOS reached the mount start path but failed at the host runtime boundary.
FUSE-T is now the active target because it is kext-less and routes FUSE through
macOS network-volume machinery, with NFS as the default backend.

GitHub-hosted macOS can install FUSE-T and can reach Operon's
`spawn_mount2_ok`, but the network volume still does not appear in the system
mount table. The latest hosted NFS diagnostic showed `mount_nfs` stuck on
`fuse-t:/operon-v014-macos-live-mount-*` while FUSE-T's local `go-nfsv4`
server listened on `127.0.0.1:<port>`. Treat hosted macOS as a diagnostic lane
until a runner image proves it can complete the FUSE-T network-volume mount.

## Required Runtime

Use FUSE-T with `pkg-config fuse` metadata available. The default backend is:

```bash
OPERON_MOUNT_MACOS_BACKEND=nfs
```

The workflow also keeps `smb` and `fskit` as explicit diagnostic options, but
the release gate should start with `nfs`.

For FUSE-T `-o` diagnostics, the mount adapter accepts comma-separated options
through:

```bash
OPERON_MOUNT_MACOS_OPTIONS=nobrowse,noattrcache
```

Use this only for FUSE-T `-o` options such as `nobrowse`, `noattrcache`,
`nonamedattr`, `rwsize=<power-of-two>`, `volname=<name>`, `nfc`, `nomtime`, or
`location=<alias>`. Raw FUSE-T arguments such as `-d` and `-l` are not accepted
through this variable because the current `fuser` integration passes custom
values as `-o <option>` entries. If the NFS listen-address option becomes
necessary, it needs explicit adapter support after confirming how FUSE-T maps
that argument through libfuse.

## Host Preflight

Run the preflight directly on a macOS host when debugging outside Actions:

```bash
OPERON_MOUNT_MACOS_BACKEND=nfs scripts/preflight-v0.14-macos-fuse-t-host.sh
```

The preflight fails early for non-macOS hosts, unsupported backend values,
missing `pkg-config fuse`, or missing FUSE-T libraries.

## Workflow Dispatch

For diagnostics, dispatch the hosted live mount workflow from the current
`main` commit:

```bash
gh workflow run "v0.14 Live Mount Smoke" \
  --repo denghongcai/Operon \
  --ref main \
  -f platform=macos \
  -f macos_runner=hosted \
  -f macos_backend=nfs \
  -f macos_options=nobrowse,noattrcache
```

For the current release gate, use a dedicated FUSE-T macOS host that passes the
preflight and can complete normal FUSE-T network-volume mounts. Register it
with labels `self-hosted`, `macOS`, and `fuse-t`, then dispatch:

```bash
gh workflow run "v0.14 Live Mount Smoke" \
  --repo denghongcai/Operon \
  --ref main \
  -f platform=macos \
  -f macos_runner=self-hosted-fuse-t \
  -f macos_backend=nfs \
  -f macos_options=nobrowse,noattrcache
```

## Success Evidence

The macOS release gate is satisfied only when the workflow completes
successfully on a macOS FUSE-T lane and the smoke output reaches:

```text
v0.14 macOS live mount smoke passed
```

Record the workflow run ID in `docs/plan/development-phases.md` and
`docs/plan/v0.14-cross-platform-live-mount.md` before publishing the release.

## Failure Handling

If the smoke fails, do not wait for repeated retries without new evidence.
Open the failed run, fetch the macOS live mount artifact, and inspect:

- `pkg-config fuse` output
- FUSE-T library and include paths
- `macOS live mount smoke exit code`
- daemon log
- mount log
- recent FUSE-T or `nfsd` process/log output
- `~/Library/Logs/fuse-t` output when FUSE-T emits user-level debug logs

If a self-hosted run is queued because no matching runner exists, cancel it and
fix runner registration before dispatching again.
