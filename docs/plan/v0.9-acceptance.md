# v0.9 Acceptance

v0.9 is scoped to endpoint model acceptance and mDNS discovery UX after the
provider abstraction was removed from user-facing configuration.

It includes:

- endpoint-only client node configuration.
- mDNS discovery as a convenience path for producing candidate endpoints.
- clear import/export behavior for discovered endpoints.
- documentation showing how Cloudflare Mesh, Tailscale, WireGuard, SSH,
  Kubernetes DNS, LAN IPs, and manual DNS names are all expressed as ordinary
  `grpc://` or `grpcs://` endpoints.
- validation that discovered endpoints do not automatically receive capability
  authorization.

It deliberately does not include provider-specific API adapters, NAT
traversal, relay network infrastructure, VPN/device mesh IP assignment, subnet
routing, global routing, or automatic capability authorization for discovered
nodes.

## Validation

The canonical v0.9 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
scripts/verify-v0.9-endpoint-model.sh
```

The endpoint model validation must prove:

- client node config accepts endpoint-only entries.
- `provider` is rejected in client node config.
- mDNS discovery can list endpoints before use.
- discovered endpoints can be exported as endpoint-only config.
- manual endpoint config remains the primary path.
- discovered nodes do not automatically receive capability authorization.

## Known Limits

- Live Cloudflare, Tailscale, WireGuard, SSH, or Kubernetes validation remains
  external to Operon. Those systems only need to make the configured endpoint
  reachable.
- mDNS discovery is local-network convenience only; it is not a provider layer.
