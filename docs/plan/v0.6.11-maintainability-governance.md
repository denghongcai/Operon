# v0.6.11 Maintainability Governance

Status: Completed.

Goal: reduce the highest-risk maintenance issues before starting larger v0.7
TUI work.

Scope:

- split low-risk daemon support areas out of [`operond/src/main.rs`](../../crates/operond/src/main.rs).
- add an explicit poisoned-lock handling path for gRPC-facing daemon state.
- make Linux-only mount support explicit at the crate dependency and CLI call
  boundary.
- add focused validation coverage for the new governance checks.

Done when:

- `operond` has separate modules for defaults, LAN advertise, store-path
  validation, status mapping, and lock handling.
- gRPC request paths return internal errors for poisoned state locks instead of
  panicking.
- non-Linux builds do not depend on [`operon-mount`](../../crates/operon-mount), and `operon mount` reports a
  clear unsupported-platform error there.
- CI runs a v0.6.11 governance validation script.
- workspace format, clippy, Rust tests, and TypeScript validation pass.

Completed:

- Split `operond` support code into `defaults`, `grpc_status`,
  `lan_advertise`, `locks`, and `store_config` modules.
- Removed direct poisoned-lock `expect` calls from [`operond/src/main.rs`](../../crates/operond/src/main.rs).
- Added a reusable gRPC lock helper that returns `Status::internal` on poisoned
  state locks.
- Changed background job/audit cleanup paths to log poisoned locks and return
  instead of panicking.
- Made [`operon-cli`](../../crates/operon-cli) depend on [`operon-mount`](../../crates/operon-mount) only for Linux targets.
- Added a non-Linux `operon mount` fallback that returns a clear unsupported
  platform error.
- Added [`scripts/verify-v0.6.11-governance.sh`](../../scripts/verify-v0.6.11-governance.sh) and wired it into CI.

Remaining:

- Larger domain splits remain future work: `operond` server/fs/job/audit
  modules, [`operon-cli`](../../crates/operon-cli) command modules, and [`operon-mount`](../../crates/operon-mount) remote/inode/FUSE
  modules.
