# v0.12.3 Windows Exec Process-Tree Cancellation

Status: Completed.

## Goal

Bring Windows non-interactive exec cancellation closer to the Unix process-group
guarantee by validating and, if practical, implementing Job Object based
process-tree termination.

## Context

v0.12.1 documented Windows exec cancellation as direct-child best-effort. That
is acceptable for an initial Windows core runtime preview, but it is a
correctness gap for long-running process trees and timeout-driven cancellation.

Unix exec already uses process-group semantics. Windows needs a platform-native
equivalent before Operon should claim parity for process-tree cancellation.

## Scope

1. Windows Job Object design.
   - Identify the smallest Windows API surface needed to create a Job Object,
     assign the spawned child process, and terminate the whole job on timeout,
     cancellation, or dropped request state.
   - Keep the abstraction local to the daemon exec runtime; do not expose
     Windows-specific types through the protocol.

2. Exec runtime integration.
   - Preserve the current Unix process-group behavior.
   - Preserve non-Windows fallback behavior where Job Objects are unavailable.
   - Make Windows cancellation semantics explicit in helper names and tests.

3. Validation.
   - Add Windows-focused tests that spawn a child process tree and verify that
     cancellation terminates descendants.
   - Add or update CI platform smoke coverage for the Windows cancellation path.
   - Keep Linux/macOS tests validating their existing guarantees.

4. Operator and protocol documentation.
   - Update `operon doctor` from best-effort warning to validated Job Object
     semantics if implementation lands.
   - Update `PROTOCOL.md`, `docs/architecture/runtime-api.md`, README platform
     notes, and the relevant validation script.

## Non-Goals

- No change to the public exec protocol shape unless the current status model
  cannot represent cancellation outcomes accurately.
- No interactive PTY session redesign.
- No Windows mount or WinFsp work.
- No claim that Windows private-file ACL semantics are fully enforced unless
  that is implemented in a separate phase.

## Acceptance Criteria

- Windows non-interactive exec cancellation either uses Job Object process-tree
  termination or documents a concrete blocker with a narrower guarantee.
- `operon doctor` reports the implemented Windows cancellation guarantee.
- CI includes focused Windows validation for the cancellation guarantee.
- Runtime API docs and `PROTOCOL.md` describe platform cancellation semantics
  without overstating parity.
- `docs/plan/development-phases.md` records the completed outcome and any
  remaining Windows exec limits.

## Completed

- Added a Windows-only Job Object guard in `crates/operond/src/exec_runtime.rs`
  that assigns spawned non-interactive exec processes to a Job Object and uses
  `TerminateJobObject` for cancellation and timeout termination.
- Preserved Unix process-group cancellation and non-Windows direct-child
  fallback behavior.
- Added Windows compile validation and a Windows-only descendant-process
  cancellation smoke test.
- Updated `operon doctor`, README, `PROTOCOL.md`, runtime API docs, CI, and
  validation scripts for the current Windows Job Object process-tree
  cancellation guarantee.
- Added `scripts/verify-v0.12.3-windows-exec-process-tree-cancellation.sh`.

## Remaining

- No v0.12.3 work remains.
- Windows private-file ACL enforcement and WinFsp mount support remain outside
  this phase.
