# v0.6.3 FS Copy Acceptance

Status: Completed.

v0.6.3 is scoped to adding same-node filesystem copy support to the protocol,
CLI, and TypeScript SDK.

It includes:

- `CopyFs(from_path, to_path)` in the gRPC runtime protocol.
- daemon-side copy within one node workspace.
- `operon fs copy <node:/from> <node:/to>`.
- `OperonClient.copyFile(nodeId, fromPath, toPath)` in the TypeScript SDK.
- `fs.copy` execution graph step support in the TypeScript SDK.
- validation for allowed copy, denied copy, and audit.

It deliberately does not include cross-node copy, recursive directory copy,
copy-on-write clone semantics, sparse-file preservation, metadata preservation,
or conflict detection.

## Contract

`CopyFs` is a daemon-side, same-node filesystem operation:

- source and target paths must resolve inside the daemon workspace.
- source requires fs read permission.
- target requires fs write permission.
- source must be a regular file.
- target parent directories must already exist.
- daemon audit records action `copy` with resource `<from_path> -> <to_path>`.
- bytes copied are returned by the daemon.

Mount/FUSE does not expose a separate copy primitive. POSIX tools such as `cp`
still appear to the mount adapter as read/write/create operations. `CopyFs` is a
protocol, CLI, and SDK convenience operation.

## Validation

The canonical v0.6.3 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
scripts/verify-v0.6.3-fs-copy.sh
```

The fs copy validation must prove:

- copying a file happens on the daemon and produces the expected target content.
- copy returns bytes copied.
- denied source read and denied target write are rejected.
- denied copy operations are audited.

## Known Limits

- Cross-node copy remains later work.
- Recursive directory copy remains later work.
- Copy does not preserve metadata beyond file bytes.
- Copy inherits the current filesystem concurrency contract: no version checks,
  locks, leases, snapshot reads, or multi-writer conflict detection.
