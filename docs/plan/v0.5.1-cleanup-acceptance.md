# v0.5.1 Cleanup Acceptance

Status: Completed.

v0.5.1 is scoped to removing the HTTP runtime facade after the gRPC migration.

It includes:

- daemon runtime API exposed through gRPC only.
- CLI as the supported human, ops, and script interface.
- TypeScript SDK as the supported programmatic interface through `nice-grpc`.
- removal of HTTP runtime handlers and hand-written HTTP CLI client code.
- documentation updates that stop presenting `http://` node endpoints as a
  supported runtime path.
- `PROTOCOL.md` for direct generated-client integration without an SDK.
- Docker and CI validation updated to use gRPC endpoints.

It deliberately does not include HTTPS/mTLS, signed node identity, Linux FUSE
mount behavior, TUI console work, agent integration, or non-LAN discovery.

## Validation

The canonical cleanup validation should run:

```bash
pnpm install --frozen-lockfile
pnpm --filter @operon/sdk proto:gen
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
pnpm --filter @operon/sdk build
scripts/verify-v0.5-docker.sh
```

The validation must prove:

- `operond` starts with a gRPC listener for runtime operations.
- `operon --json` provides the scriptable machine-readable interface.
- fs read/write streaming works through gRPC.
- job logs and stdin work through gRPC.
- policy denials and auth failures still produce stable client errors.
- audit records are still produced for runtime operations.
- TypeScript SDK calls gRPC through `nice-grpc`.

## Completion Notes

- Removed the axum HTTP runtime facade from `operond`.
- Removed the hand-written HTTP client path from [`operon-cli`](../../crates/operon-cli).

- Kept the TypeScript SDK on `nice-grpc` and removed fetch-based fallback

## Known Limits

- A minimal process or load-balancer health endpoint may be reconsidered later,
  but it must not become a parallel runtime API.
- Direct `curl` access is intentionally not a supported product surface; use
  `operon --json` instead.
