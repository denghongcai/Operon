# v0.10.1 Filesystem Consistency and Workspace Hardening

Status: Completed.

## Purpose

Define the first concrete filesystem consistency contract before v1.0 and
reduce Linux workspace containment risk without turning Operon into a file sync
or distributed lock system.

## Scope

- Add opaque filesystem versions to `StatFs`, `ListFs`, `WriteFile`, and
  `CopyFs` responses.
- Add optional mutation preconditions for filesystem writes and destructive
  mutations:
  - `expected_version` rejects stale writes with gRPC `FAILED_PRECONDITION`.
  - `require_absent` lets clients express create-if-absent behavior.
- Expose guarded writes through the Rust CLI and TypeScript SDK.
- Keep versions opaque. Clients compare and echo them back; they must not parse
  inode, timestamp, or size details.
- Harden Linux workspace containment with fd-relative `openat2` checks using
  `RESOLVE_BENEATH` when the kernel supports it, while keeping the existing
  canonical containment fallback for unsupported kernels and non-Linux hosts.
- Document the consistency contract in `PROTOCOL.md` and
  `docs/architecture/runtime-api.md`.
- Add validation coverage in
  [`scripts/verify-v0.10.1-fs-consistency-workspace-hardening.sh`](../../scripts/verify-v0.10.1-fs-consistency-workspace-hardening.sh).

## Completed

- `proto/operon/runtime.proto` now carries `FsPrecondition`, filesystem
  versions, and mutation precondition fields.
- `crates/operond/src/fs_service.rs` computes opaque versions, validates
  preconditions, returns `FAILED_PRECONDITION` on stale writes, and tests
  version/precondition behavior.
- `crates/operon-fs` declares Linux `openat2(RESOLVE_BENEATH)` containment as
  the active Linux hardening strategy and validates fd-relative resolution where
  available.
- `crates/operon-grpc-client`, `crates/operon-cli`, and
  `packages/sdk-js` can send guarded write requests.
- Runtime/package versions are aligned to `v0.10.2` / `0.10.2` because the
  protocol shape changed again during the v0.10.1/v0.10.2 work.
