# v0.9.7 Runtime API Hardening

Status: Completed.

Goal: close review findings around runtime API pagination, streaming API
documentation, SDK memory behavior, and daemon-side job validation.

## Scope

1. Filesystem list pagination.
   - Add `page_size`, `page_token`, and `next_page_token` to the gRPC
     `ListFs` path.
   - Keep existing CLI and SDK helpers returning the complete directory listing
     by following pages internally.

2. Runtime API documentation alignment.
   - Document bidirectional service tunnel RPCs explicitly.
   - Make list pagination wording match the concrete protocol surface.

3. SDK streaming memory behavior.
   - Stream `ReadableStream` bodies into gRPC chunks without concatenating the
     full body first.

4. Job request validation.
   - Reject daemon job requests where both `command` and `argv` are empty.

## Done When

- Rust protocol conversions preserve fs list pagination metadata.
- Daemon `ListFs` returns deterministic pages and rejects invalid page tokens.
- Rust CLI, mount adapter, and TypeScript SDK callers continue to collect full
  listings for their current public helpers.
- TypeScript SDK write helpers do not buffer `ReadableStream` bodies before
  creating gRPC chunks.
- Daemon rejects empty job requests before spawning a shell.
- README, `PROTOCOL.md`, runtime API docs, AGENTS guidance, and phase tracker
  mention the v0.9.7 cleanup.
