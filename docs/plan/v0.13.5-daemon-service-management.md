# v0.13.5 Daemon Service Management

Status: Completed.

## Goal

Give `operond` a first-class managed daemon setup path without adding a
traditional `operond start --background` mode.

## Context

`operond start` is currently a foreground service process. That is still the
right primitive for development, Docker, systemd, launchd, Windows Service
wrappers, and CI because logs, signals, restart policy, and working directory
are owned by the process supervisor.

For real users, however, "run the daemon" should not require hand-writing
systemd units, launchd plists, or Windows Service registration commands. Operon
should provide a system-service management surface that installs and controls
the platform-native supervisor entry while leaving actual background process
ownership to the operating system.

## Scope

1. Command surface.
   - Add `operond service install --config <path>`.
   - Add `operond service start`.
   - Add `operond service stop`.
   - Add `operond service status`.
   - Add `operond service uninstall`.
   - Keep `operond start` as the foreground runtime command.

2. Platform backends.
   - Linux: support user-level systemd units first.
   - macOS: support launchd user agents.
   - Windows: implement Windows Service registration through a native service
     entrypoint that speaks the Service Control Manager protocol.

3. Install behavior.
   - Generate service definitions that invoke the current `operond start
     --config <path>` binary path.
   - Preserve explicit config paths; do not hide config generation inside
     service install.
   - Avoid storing secrets in service definitions. Existing config token file
     and token env behavior remain the supported secret paths.
   - Surface clear errors for missing config files, non-absolute executable
     paths, unsupported platforms, and supervisor command failures.

4. Docs and diagnostics.
   - Document service management in README and DEVELOPMENT.
   - Add `operon doctor` or `operond service status` guidance that explains
     supervisor state and common failures where practical.
   - Keep `operond start --help` free of `--background` or daemonize flags.

5. Validation.
   - Add unit tests for rendered systemd and launchd service definitions.
   - Add CLI/help validation for the service subcommands.
   - Add platform-gated smoke validation where CI can safely exercise the
     supervisor, and static validation where it cannot.

## Non-Goals

- No `operond start --background` or self-daemonizing fork mode.
- No custom process supervisor, restart loop, or pidfile manager inside
  `operond`.
- No replacement for Docker, Kubernetes, systemd, launchd, or Windows Service
  process ownership.
- No config generation redesign; onboarding and config files remain separate
  surfaces.

## Acceptance Criteria

- Users have a documented `operond service ...` path for installing, starting,
  stopping, checking, and uninstalling the daemon through platform-native
  supervision.
- `operond start` remains a foreground process and has no `--background`
  option.
- Linux and macOS service definitions are generated from explicit executable
  and config paths with tests covering their rendered content.
- Windows support is implemented through a real Service Control Manager
  entrypoint and validated at least with target compilation and static command
  coverage.
- README, DEVELOPMENT, AGENTS, and `docs/plan/development-phases.md` describe
  the service-management model consistently.

## Completed

- Added `operond service install/start/stop/status/uninstall` as the daemon
  service-management command surface.
- Kept `operond start` as the foreground runtime command and kept
  `operond start --help` free of `--background`.
- Implemented Linux user-level systemd service installation and control.
- Implemented macOS launchd user-agent installation and control.
- Generated service definitions invoke `operond start --config <path>` with an
  explicit config path and do not embed token values or other secrets.
- Implemented Windows Service registration through `operond service run
  --config <path>`, a hidden entrypoint that connects to the Service Control
  Manager, reports service status, and maps SCM stop/shutdown controls to daemon
  shutdown.
- Added unit coverage for the service command surface, rendered systemd unit,
  rendered launchd plist, and Windows Service registration command.
- Added `scripts/verify-v0.13.5-daemon-service-management.sh` and wired it
  into the consolidated validation runner.
- Added platform smoke CI coverage for daemon service-management tests on
  macOS and Windows.
- Updated README, DEVELOPMENT, AGENTS, and this phase tracker.

## Remaining

- No v0.13.5 daemon service management work remains.
- Elevated Windows install/start/stop smoke remains a manual pre-release check
  when practical because CI runners may not allow persistent service
  registration.
