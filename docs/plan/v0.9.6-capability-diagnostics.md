# v0.9.6 Capability Diagnostics

Status: Completed.

Goal: expose policy decision diagnostics through the runtime, CLI, and SDK so
users and agents can ask why a capability action is allowed or denied before
running it.

This phase follows v0.9.5 policy language hardening. It turns the shared
`PolicyDecision` vocabulary into an operator-facing diagnostic surface without
adding new capability types or changing the existing `policy:` config schema.

## Scope

1. Runtime diagnostic RPC.
   - Add a small unary gRPC request that asks the daemon to explain one
     capability action.
   - Return the same subject, capability id, action, resource, allowed flag,
     reason code, and message vocabulary used by internal policy decisions.
   - Keep the daemon as the source of truth for remote policy diagnostics.

2. Supported diagnostic families.
   - Filesystem: explain `read`, `write`, and `delete` decisions for a path.
   - Job: explain `run` decisions for a working directory and optional timeout.
   - Secret: explain `use` decisions for a secret name.
   - Service: explain `check` and `forward` decisions for a service id.
   - Unsupported capability/action pairs should return a denied diagnostic with
     `unsupported-action`.

3. CLI UX.
   - Add `operon capability explain <node> <capability_id> <action> <resource>`.
   - Add optional `--timeout-secs` for job run diagnostics.
   - Support `--json` with stable machine-readable output.
   - Human output should be compact and name the reason code.

4. SDK UX.
   - Add a TypeScript SDK helper for capability diagnostics.
   - Keep the helper close to existing capability/service/job helpers.

5. Documentation and validation.
   - Document the runtime request/response in `PROTOCOL.md` and runtime API
     docs.
   - Update README CLI examples where relevant.
   - Add a dedicated validation script and CI matrix entry for v0.9.6.
   - Update agent guidance in `AGENTS.md`.

## Non-Goals

- No new capability type.
- No clipboard, screen, audio, remote desktop, file sync, VPN, mesh, NAT
  traversal, relay, routing, or packet policy.
- No replacement of the current `policy:` config schema.
- No general-purpose policy expression language.
- No local CLI reimplementation of remote daemon policy.

## Done When

- The active gRPC protocol exposes capability diagnostics.
- Daemon diagnostics are derived from the same policy decision helpers used by
  runtime authorization paths.
- `operon capability explain` works for filesystem, job, secret, service, and
  unsupported action diagnostics.
- `operon capability explain --json` emits stable fields for agents and
  scripts.
- TypeScript SDK exposes the same diagnostic shape.
- Docs, validation script, and CI are updated.
