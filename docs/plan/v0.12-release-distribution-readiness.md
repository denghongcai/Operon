# v0.12 Release / Distribution Readiness

Status: Completed.

Goal: make the public release surface match the platform decisions made in
v0.11.3, or explicitly narrow the pre-1.0 release target set so users do not
infer unsupported macOS or Windows parity.

## Scope

This phase should turn the v0.11.3 platform capability matrix into release
policy and release automation.

Planned:

1. Release target decision.
   - Decide whether pre-1.0 public releases remain Linux-only or expand to
     macOS and Windows core runtime preview binaries.
   - If expanding, publish only the core daemon and CLI runtime surface:
     daemon startup, CLI commands, gRPC protocol, auth/config, audit, trace,
     graph, capability diagnostics, filesystem RPCs, exec, and service
     forwarding.
   - Keep Linux FUSE as the only mount adapter; do not claim macFUSE or WinFsp
     support.

2. Release workflow alignment.
   - If expanding, add release matrix entries for selected macOS and Windows
     targets:
     - `macos-x86_64`
     - `macos-aarch64`
     - `windows-x86_64`
   - Keep Linux targets:
     - `linux-x86_64`
     - `linux-arm64`
     - `linux-armv7`
   - Package Linux and macOS binaries as `.tar.gz`.
   - Package Windows binaries as `.zip`.
   - Generate checksums for every archive.

3. Release smoke validation.
   - Run built `operon --version` and `operond --version` for every native
     release target.
   - Run a minimal CLI help smoke for `operon --help`, `operon doctor --help`,
     and `operon exec --help`.
   - Keep GLIBC baseline validation Linux-only.
   - Keep mount validation Linux-only.

4. Documentation and release validation.
   - Update README install instructions and Project Status to match the chosen
     target set.
   - Update `docs/architecture/technology-and-protocol-decisions.md` if the
     supported public target set changes.
   - Update `scripts/verify-readme-quickstart-docker.sh` when README
     Quickstart, release packaging, or install prerequisites change.
   - Add or update release validation so CI can detect drift between docs,
     release workflow targets, and public artifact naming.

## Non-Goals

- No macFUSE or WinFsp implementation.
- No Windows Job Object process-tree cancellation implementation unless a
  release smoke requires narrowing a claim.
- No new PTY backend beyond `portable-pty`.
- No network, VPN, mesh, NAT traversal, or relay responsibility expansion.
- No package manager publishing in this phase unless the binary archive
  release flow is already stable.

## Done When

- The repository clearly states whether pre-1.0 public releases are Linux-only
  or include macOS/Windows core runtime preview binaries.
- Release workflow targets, README install instructions, architecture docs, and
  validation scripts agree on the supported artifact set.
- Every released daemon/CLI archive has a checksum.
- Every native release target runs basic version/help smoke validation.
- Linux-only mount and GLIBC baseline constraints remain explicit.

## Completed

- Chose to expand pre-1.0 public artifacts to Linux plus macOS/Windows core
  runtime preview binaries.
- Split draft release Rust automation into Linux container builds and native
  macOS/Windows builds.
- Kept Linux release targets for `linux-x86_64`, `linux-arm64`, and
  `linux-armv7`.
- Added native preview release targets for `macos-x86_64`, `macos-aarch64`,
  and `windows-x86_64`.
- Kept Linux and macOS archives as `.tar.gz`; packaged Windows archives as
  `.zip`.
- Added native daemon/CLI smoke validation for `--version`, `operon --help`,
  `operon doctor --help`, and `operon exec --help`.
- Kept GLIBC baseline validation Linux-only.
- Updated README install instructions, Project Status, architecture docs, and
  README quickstart validation target detection.
- Added `scripts/verify-v0.12-release-distribution-readiness.sh` and wired it
  into CI and DEVELOPMENT.md.
- Bumped Rust crate versions, TypeScript SDK package version, CLI version, and
  runtime health `PROTOCOL_VERSION` to `0.12.2` / `v0.12.2` so public release
  surfaces stay aligned after the v0.12 through v0.12.2 work.

## Remaining

- No v0.12 work remains.
- Linux FUSE remains the only supported mount adapter.
