# v0.3 Acceptance

v0.3 makes the v0.2 runtime more usable on a local network while preserving the project boundary: Operon discovers and uses reachable endpoints, but it does not create a VPN, relay, NAT traversal layer, or mesh IP network.

Historical note: this file describes the v0.3 implementation at the time it was
accepted. Some commands and API shapes in this snapshot were replaced by later
phases; the current runtime uses unified `config.yaml`, gRPC, `PROTOCOL.md`,
and the latest completed phase entries.

## Acceptance Scope

- Filesystem raw read/write endpoints stream daemon-side IO instead of loading whole files into memory.
- Jobs expose a streaming log endpoint and stdin write/close endpoints.
- CLI supports `--json` and `--quiet` output modes for core commands.
- CLI can generate starter node and policy files with `operon init config` and `operon init policy`.
- Daemon reloads completed job records from the JSONL store and exposes `job list`.
- CLI exposes `job list`, `audit show`, and `trace list`.
- LAN discovery is limited to mDNS through `operon node discover --timeout-secs 3`.
- Mount work remains a read-only PoC through one-shot local materialization plus a manifest.

## Canonical Validation

```bash
pnpm install --frozen-lockfile
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm -r test
scripts/verify-v0.5-docker.sh
```

## Known Limits

- v0.3 originally used HTTP streaming. The current runtime uses gRPC streaming.
- Job stdin is write/close based, not a full interactive TTY.
- LAN mDNS discovery returns endpoint records only and does not grant capability access.
- The store remains append-only JSONL with narrow query APIs.
- The mount PoC is not FUSE / WinFsp and does not provide live sync.
