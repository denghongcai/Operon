# v0.18.9 Windows Runner Image Migration Smoke

Status: Completed.

## Goal

Make GitHub-hosted Windows runner image migration an explicit release gate by
validating Operon's Windows CI, live mount, release build, and downloaded
release smoke paths on the Windows Server 2025 hosted image.

## Context

Before this phase, workflows still used `windows-latest` in the main CI
platform smoke, cross-platform live mount smoke, draft release build, release
artifact verification, and downloaded release install usability workflows. The
latest pre-phase main CI evidence showed `windows-latest` already resolving to
Microsoft Windows Server 2025, image `windows-2025`, version `20260510.128.1`;
the Windows platform smoke passed on commit
`65dfb9d733e4a273e54d5ee7f8f2fa3032003f5e`.

That pass proved the normal CI Windows smoke worked on the migrated image, but
it did not make the migration decision durable. This phase moves
release-critical jobs away from the floating label, updates validation scripts
that preserved stale `windows-latest` assertions, and adds targeted Windows
Server 2025 evidence before the next public release.

## Scope

1. Windows runner inventory and label policy.
   - Inventory every workflow and validation script that names
     `windows-latest`: `.github/workflows/ci.yml`,
     `.github/workflows/live-mount-smoke.yml`,
     `.github/workflows/release-draft.yml`,
     `.github/workflows/verify-release-artifacts.yml`,
     `.github/workflows/verify-release-install-usability.yml`, and existing
     phase validation scripts that assert those labels.
   - Record the chosen policy in this phase: migrate release-critical Windows
     jobs to explicit `windows-2025` once targeted smoke evidence passes,
     while only keeping `windows-latest` where the repo intentionally wants a
     floating future-image signal.
   - Update validation scripts that hard-code `windows-latest` so they check
     the chosen label policy instead of preserving stale assertions.

2. Targeted Windows Server 2025 smoke workflow.
   - Add a manual workflow, tentatively
     `.github/workflows/windows-runner-image-smoke.yml`, defaulting to
     `windows-2025` with an optional comparison input for `windows-latest`.
   - Print runner evidence at the start of the job: operating system,
     `ImageOS` / `ImageVersion` where available, `RUNNER_OS`, PowerShell
     version, Rust version, Git version, and WinFsp installation state.
   - Install the same Windows prerequisites used by release jobs:
     Rust stable, protoc `25.3`, and WinFsp via Chocolatey.
   - Run the Windows-sensitive checks from the normal platform smoke:
     workspace check, protocol/config/filesystem core tests, mount adapter
     tests, exec policy tests, daemon PTY smoke, Windows Job Object
     cancellation, daemon service management, Windows private-file ACL model
     tests, and CLI session dimension tests.
   - Build release binaries for `x86_64-pc-windows-msvc` and smoke-test
     `operon.exe --version` and `operond.exe --version`.

3. Release-path Windows smoke.
   - Run `scripts/verify-release-artifacts.sh <tag> <repo>` on the targeted
     Windows runner against the current public release tag used by existing
     post-download validation, currently `v0.16.6`.
   - Run `scripts/verify-release-install-usability.sh <tag> <repo>` and
     `scripts/verify-release-service-management-smoke.sh <tag> <repo>` on the
     targeted Windows runner so downloaded archive extraction, `PATH`
     invocation, and fake `sc.exe` service-management behavior are covered on
     Windows Server 2025.
   - Keep Linux glibc, macOS, and Alpine/musl decisions unchanged; this phase
     only proves the Windows hosted-image path.

4. Existing workflow migration.
   - After the targeted Windows Server 2025 smoke passes, update operational
     Windows jobs to the chosen label policy in:
     `.github/workflows/ci.yml`, `.github/workflows/live-mount-smoke.yml`,
     `.github/workflows/release-draft.yml`,
     `.github/workflows/verify-release-artifacts.yml`, and
     `.github/workflows/verify-release-install-usability.yml`.
   - Keep the release draft workflow tag-triggered and non-publishing until the
     normal release process runs; do not add a path that publishes or edits a
     GitHub Release from the migration smoke workflow.
   - Preserve the existing WinFsp installation behavior unless the Windows
     Server 2025 smoke shows a real break.

5. Documentation and focused validation.
   - Update `docs/quality/release-ci-observability.md` with the Windows runner
     image smoke, expected evidence, and release-triage guidance.
   - Update README or DEVELOPMENT only if the maintainer-facing release
     commands or required release gates change.
   - Update `AGENTS.md` and `docs/plan/development-phases.md` with planned and
     completed phase state.
   - Add `scripts/verify-v0.18.9-windows-runner-image-migration-smoke.sh` to
     validate the phase doc, workflow wiring, runner label policy, docs, and
     phase tracker.
   - Wire the focused validation through the consolidated `core` validation
     group in `scripts/ci/run-validations.sh`.

## Non-Goals

- No Windows feature rewrite, mount adapter rewrite, service-management
  rewrite, or PTY backend change.
- No release publication, draft release creation, tag push, or GitHub Release
  mutation from the migration smoke workflow.
- No change to Linux glibc baseline, Alpine/musl support policy, macOS FUSE-T
  behavior, or public release archive naming.
- No installer UI, code signing, notarization, MSI, Chocolatey, winget, or
  automatic update work.
- No Node 24 / JavaScript action-runtime cleanup unless it blocks the Windows
  runner image smoke; action-runtime hardening remains a separate CI phase.

## Acceptance

- The repo has a durable Windows runner label policy instead of relying on
  stale `windows-latest` assertions.
- A targeted GitHub-hosted Windows Server 2025 smoke run proves the Windows
  platform tests, WinFsp-dependent mount adapter checks, release binary build,
  release artifact verifier, downloaded install usability check, and downloaded
  service-management smoke.
- Operational Windows workflow jobs are aligned with the chosen label policy.
- Release documentation and CI observability guidance explain where to find
  runner image evidence and how to triage a Windows image migration failure.
- Focused validation and the consolidated `core` validation group pass after
  the workflow/docs updates.
