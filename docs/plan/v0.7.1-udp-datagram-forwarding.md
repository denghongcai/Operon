# v0.7.1 UDP / Datagram Forwarding

v0.7.1 is scoped to UDP/datagram forwarding as a separate protocol and product
phase from v0.7 TCP service forwarding.

It adds datagram support without tunneling UDP through the existing
`OpenServiceTunnel` byte stream because UDP needs packet-boundary semantics.

It should include:

- explicit UDP service policy entries.
- `OpenServiceDatagramTunnel` as a datagram-oriented gRPC protocol.
- packet-boundary preservation.
- local UDP listener support in the CLI.
- TypeScript SDK datagram tunnel support over `nice-grpc`.
- audit events for UDP forwarding start, packet flow summaries, and failure
  conditions.
- validation with a local UDP echo service.

It deliberately should not include NAT traversal, UDP hole punching, relay
networking, global service routing, arbitrary target host/port forwarding, mDNS
relay behavior, or QUIC transport replacement.

## Design Decisions

- `ServiceProtocol` supports both `tcp` and `udp`.
- One local UDP peer maps to one daemon-side UDP socket session.
- The client-side forwarder assigns stable `peer_id` values for local UDP peer
  addresses.
- The daemon expires idle peer sessions after a bounded timeout.
- Audit records forwarding activity at tunnel start rather than every datagram.
- Oversized datagrams are rejected per peer with a close event.

## Proposed Validation

The canonical v0.7.1 validation should run:

```bash
cargo fmt --check
cargo check --workspace --locked
cargo test --workspace --locked
cargo clippy --workspace --locked -- -D warnings
pnpm typecheck
pnpm test
scripts/verify-v0.7.1-udp-datagram-forwarding.sh
```

The UDP validation should prove:

- the daemon can list and health-check a policy-configured UDP service if UDP
  health checks are included in scope.
- the CLI can bind a local UDP socket and forward datagrams to that service.
- datagram boundaries are preserved end to end.
- unknown or policy-denied UDP services are rejected.
- audit output records UDP forwarding activity.
